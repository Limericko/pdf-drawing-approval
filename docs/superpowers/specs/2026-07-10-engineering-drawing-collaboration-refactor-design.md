# 工程图纸协同平台整体重构设计

- 日期：2026-07-10
- 状态：设计已确认，待书面审阅
- 当前版本：0.9.2
- 重构路径：分阶段垂直重构
- 目标部署：中国香港云区域，B+ 高可用容器化架构

## 1. 结论

本项目从“Windows 局域网 PDF 图纸审批系统”重构为“工程图纸协同平台”。审批仍是核心，同时统一图纸提交、PDF 审阅、批注问题、签章、PDM 零件版本、WebDAV 同步、打印归档和系统运维。

重构不采用一次性推倒重写。保留 React、TypeScript、Express、Electron 和现有业务规则，按完整业务链逐段替换 UI、模块边界、数据存储和部署方式。最终运行在香港云区域，所有办公室和家庭用户通过同一 HTTPS 域名直接登录，不保留本地业务服务器，也不建设本地与云端的数据同步。

目标运行架构为：

- CDN/WAF 和云负载均衡。
- 两个无状态应用实例，支持故障摘除和滚动发布。
- 独立后台 Worker，负责 PDF、通知、扫描和 WebDAV 任务。
- 托管 PostgreSQL 高可用数据库。
- 私有 S3 兼容对象存储。
- WebDAV 受控双向同步。
- 结构化日志、指标、告警、审计和恢复演练。

业务代码保持模块化单体。不引入微服务和 Kubernetes。

## 2. 当前项目审计

### 2.1 已有能力

当前系统已覆盖完整的受控图纸流程：

- 设计师网页或 Electron 客户端提交 PDF。
- 主管与工艺并行审核。
- PDF 预览、几何批注、评论和操作时间线。
- 设计师、主管、工艺三方签名定位与签后 PDF。
- Windows 客户端打印并归档。
- 图纸台账、批量处理、异常修复和版本追溯。
- PDM 零件、图纸版本、项目使用关系和元数据补录。
- 用户、权限、通知、备份、清理、日志和风险看板。
- Electron 客户端、服务端 EXE、托盘助手和 Windows 安装包。

因此本次重构的重点不是补齐一个最小产品，而是整理已经增长成平台的职责、信息架构和运行边界。

### 2.2 当前技术结构

- 前端：React 19、Vite、TypeScript、全局 CSS、Lucide。
- 后端：Express、TypeScript、Zod、内置 `node:sqlite`。
- PDF：PDF.js、pdf-lib、fontkit。
- 桌面端：Electron；另有 Tauri 托盘助手。
- 测试：Vitest、Supertest。
- 部署：Windows 服务端 EXE、客户端安装包和局域网发布目录。

### 2.3 主要结构问题

现状功能可靠，但增长集中在少数大文件：

- `src/client/styles.css` 约 6180 行，承担全站视觉和响应式规则。
- `ApprovalDetailPage.tsx` 约 1457 行，混合数据加载、PDF 工作台、审核、打印和弹层状态。
- `api.ts` 约 1150 行，同时定义大量类型和所有领域请求。
- `SettingsPage.tsx`、`OperationsTab.tsx`、`approvals.ts` 等文件职责过宽。
- 前端采用手写 hash 路由和页面级局部状态，跨页面任务和通知缺少统一模型。
- 目前 UI 测试大量依赖读取源码或 CSS 字符串，能够保护结构约定，但不能代替真实浏览器交互和视觉验证。
- SQLite、Windows 文件路径、服务端 EXE 和本地目录管理与云端高可用目标不匹配。

### 2.4 当前基线

设计阶段已完成以下只读或生成物验证：

- `npm run build` 通过。
- 核心审批、PDM、权限、路由和布局相关的 11 个测试文件、132 项测试通过。
- 构建提示 PDF 相关 chunk 超过 500 kB，需要在重构中继续做异步加载和缓存边界。
- 工作树在设计文档写入前保持干净。

## 3. 已确认决策

### 3.1 产品与范围

- 产品定位为工程图纸协同平台。
- 第一优先场景是 PDF 审阅与标注工作台。
- 全站采用“精密工业”视觉方向。
- PDF 工作台采用文档优先三栏式。
- 批注采用“普通说明 + 正式问题”的混合模式。
- 正式问题由处理人提交待复核，创建人或对应审核角色关闭。
- 管理员强制关闭必须填写原因并审计。

### 3.2 访问与部署

- 所有用户统一访问香港云端系统。
- 使用 HTTPS 域名直接登录，不要求客户端安装 VPN。
- 不保留办公室本地服务端，不建设离线双系统。
- 公网关闭开放注册，账号由管理员邀请或创建。
- 使用 B+ 高可用架构，接受持续云资源费用。
- 使用中型容量基线：30 至 100 个账号，10 至 30 人并发。
- 首次正式迁移允许 1 至 2 小时计划维护窗口。

### 3.3 WebDAV

- 云端与 WebDAV 双向同步。
- 云端数据库和对象存储是唯一权威。
- WebDAV 新文件可以受控导入。
- 云端已批准版本可以发布到 WebDAV。
- 不允许同路径不同内容静默覆盖。
- 默认不双向传播删除。

## 4. 目标与非目标

### 4.1 目标

- 让用户从统一任务入口完成提交、审核、修改、复核、补录和同步异常处理。
- 让图纸、审批、零件版本、问题、文件和同步记录互相可追溯。
- 显著提升连续审图、标注、定位问题和提交审核结论的效率。
- 建立清晰的前端领域模块、后端服务边界和数据契约。
- 支持香港云端公网安全登录和中型团队并发。
- 支持单应用实例故障接管和无计划停机的滚动发布。
- 将 SQLite 和本地文件目录迁移到 PostgreSQL 与对象存储。
- 用真实浏览器测试、集成测试、性能测试和恢复演练保护重构。

### 4.2 非目标

本次重构不包含：

- 完整 PLM、BOM、工程变更单和供应商门户。
- 面向客户或供应商的多租户体系。
- 开放自注册。
- 办公室断网时的离线审批。
- 本地与云端双写数据库。
- 微服务、Kubernetes、多区域主动双活。
- WebDAV 删除自动传播或冲突自动覆盖。
- 为了视觉效果更换 React、Express 或 Electron。

## 5. 目标系统架构

### 5.1 边缘与访问层

公网请求依次经过：

1. 域名、CDN 和 WAF。
2. HTTPS、限流、安全响应头和基础攻击防护。
3. 香港云负载均衡。
4. 健康的应用实例。

Electron 客户端加载同一 Web 应用，只保留本地打印、自动更新和必要的系统能力桥接。浏览器和 Electron 不使用两套业务实现。

### 5.2 应用运行层

两个应用实例运行相同的签名 Docker 镜像，并保持无状态。会话、任务和业务状态不能只存在于单机内存或本地磁盘。

应用实例包含：

- React 静态资源服务。
- Express API。
- 模块化业务层。
- 健康检查和指标端点。

后台 Worker 独立运行：

- PDF 有效性检查。
- 签章和审查版生成。
- 恶意文件扫描。
- 邮件与站内通知。
- WebDAV 导入、发布、校验和重试。
- 维护、迁移和失败任务处理。

Worker 使用 PostgreSQL 中的任务、Outbox 和租约记录协调。当前规模不额外引入 Redis 或独立消息中间件，除非实施阶段的压测证明有必要。

### 5.3 模块化单体边界

后端按领域组织：

- 身份与权限。
- 审批工作流。
- 文档与存储。
- 批注与问题。
- 签章与渲染。
- PDM 零件与版本。
- WebDAV 同步。
- 通知中心。
- 运维与审计。

模块间通过明确服务接口和领域事件协作。HTTP 路由只处理协议、认证、解析和响应；业务规则进入服务层；数据库访问进入仓储层。

### 5.4 数据层

PostgreSQL 是业务状态唯一真相：

- 所有状态变化在事务中完成。
- 关键唯一性、外键和检查规则由数据库约束保护。
- 业务事件通过事务 Outbox 可靠交给 Worker。
- 高并发修改使用版本号或条件更新检测冲突。

对象存储是文件权威库：

- 原始 PDF、审查版、签后 PDF 和签名资源使用稳定对象键。
- 存储桶保持私有并开启服务端加密和版本化。
- 数据库保存对象键、哈希、大小、类型和生成状态。
- 文件发布后不可在原对象上静默覆盖。

## 6. 信息架构

### 6.1 统一应用外壳

所有角色使用同一导航骨架，权限决定可见模块和可执行操作。导航分组为：

- 工作：首页、我的任务、问题中心。
- 图纸：提交图纸、图纸中心、批量处理。
- PDM：零件与版本、元数据待办。
- 协同与系统：WebDAV 同步、系统管理。
- 个人：资料、安全、通知偏好和退出。

侧栏支持展开和收起；窄屏改为稳定的顶部/底部主导航和页面内二级入口。页面标题、面包屑和返回路径保持一致。

### 6.2 角色首页

设计师优先看到：

- 需修改的问题。
- 待补录 PDM 元数据。
- 提交、签名和发布状态。
- 最近图纸。

主管优先看到：

- 待我审核。
- 待复核问题。
- 即将超期任务。
- 跨项目审阅队列。

工艺优先看到：

- 工艺待审。
- 材料和工艺问题。
- 待复核修改。
- 版本变化提醒。

管理员优先看到：

- 服务、安全和备份告警。
- WebDAV 冲突。
- 失败的 PDF 与同步任务。
- 用户、权限和审计异常。

首页不使用装饰性大卡片堆叠。任务列表按阻塞程度、严重级和期限排序。

### 6.3 主业务闭环

主流程为：

1. 网页上传或 WebDAV 导入。
2. 隔离、扫描、PDF 校验、哈希去重和元数据识别。
3. 签名位置与提交信息确认。
4. 主管和工艺并行审核。
5. 说明交流或正式问题处理。
6. 处理人提交待复核，审核人关闭问题。
7. 审批通过后异步生成签章和审查产物。
8. 发布 PDM 不可变图纸版本。
9. 发布到 WebDAV 指定目录并校验回读。
10. Electron 本地打印或受控归档。

## 7. 视觉与组件系统

本节定义总体方向。组件级令牌、目录结构、状态矩阵、响应式规则、旧样式映射、迁移阶段和视觉回归门禁以同目录的 `2026-07-10-ui-design-system-design.md` 为实施依据。

### 7.1 视觉方向

采用“精密工业”方向：

- 石墨色用于导航、工具区和稳定框架。
- 浅灰用于工作面和分区。
- 白色用于 PDF、表格和需要聚焦的编辑面。
- 青绿色用于主操作、选择态和成功路径。
- 红色只用于正式问题、危险和阻断。
- 琥珀色用于说明、提醒和同步异常。

不使用单一蓝色主题、紫色渐变、装饰性光斑、过大圆角、卡片嵌套或营销式大标题。

### 7.2 设计令牌

建立共享令牌：

- 颜色、文字、边框、状态和图表颜色。
- 4/8 像素间距体系。
- 控件与面板圆角不超过 8 像素。
- 明确的阴影、焦点和 z-index 层级。
- 稳定的工具栏、图标按钮和表格行尺寸。
- 中文优先的字体栈和等宽/等宽数字规则。

现有 6000 行全局 CSS 逐步拆为：

- 基础令牌和重置。
- 应用外壳。
- 共享组件。
- PDF Studio。
- 各领域页面样式。

### 7.3 组件原则

- 使用图标按钮表示熟悉的工具动作，并提供 tooltip 和可访问名称。
- 状态使用统一标签，不在每个页面重复定义颜色和文案。
- 加载使用与布局匹配的骨架屏。
- 空状态提供下一步动作。
- 错误显示在发生位置并提供可恢复操作，不使用 `window.alert()`。
- 表格、筛选、分页和批量操作使用统一组件。
- 页面区块不包装成层层卡片。
- 手机端保持查看、讨论和处理状态能力；复杂精确绘图保留桌面端。

## 8. PDF 审阅与标注工作台

### 8.1 布局

采用文档优先三栏式：

- 左栏：页面缩略图、页码、每页问题数量和快速定位。
- 中栏：PDF 画布、标注工具和视口控制。
- 右栏：问题、说明、图纸属性和所选对象属性。
- 顶栏：图纸上下文、绘图工具、撤销和重做。
- 底栏：保存状态、页码、缩放和审核结论。

“通过并打开下一张”保留任务流优势，但不牺牲三栏布局。

### 8.2 工具

支持：

- 选择和平移。
- 定位点、矩形、圆形、箭头。
- 文字、画笔和云线。
- 移动、缩放、改色、复制和删除。
- 撤销、重做和连续标注。
- 适宽、适页、缩放、页码、缩略图和全屏。

绘图后在画布附近打开上下文编辑器，选择“说明”或“问题”，避免先填写表单再回画布定位。

### 8.3 混合批注

普通说明包含：

- 图形和页面坐标。
- 内容、作者和时间。
- 讨论和定位。

正式问题额外包含：

- 严重级。
- 负责人。
- 到期时间。
- 状态。
- 处理与复核记录。

问题状态为：

`待处理 -> 处理中 -> 待复核 -> 已关闭`

设计师或负责人不能直接关闭自己的问题。问题创建人或对应审核角色负责复核关闭；管理员强制关闭必须填写原因。

未关闭的高严重级问题阻止审批通过。中低级问题明确提醒，并按权限允许继续。

### 8.4 保存与并发

- UI 显示保存中、已保存和保存失败。
- 网络中断时保留本地未提交草稿并明确提示。
- 恢复网络后通过幂等键重试，避免重复问题。
- 服务端向在线用户推送新问题和状态变化。
- 编辑使用版本号检测冲突，不允许最后写入者无提示覆盖。

### 8.5 性能

- PDF Worker 异步解析。
- 页面虚拟化，只渲染可见页和邻近页。
- 缩略图延迟加载。
- 请求取消和路由切换清理。
- PDF 与缩略图分块缓存。
- 标注使用标准化坐标，与缩放和屏幕尺寸解耦。
- 大型 PDF 和批量操作进入后台任务并显示进度。

## 9. WebDAV 双向同步

### 9.1 定位

WebDAV 是受控文件交换端，不是第二套业务数据库。审批状态、PDM 版本、问题和同步决策只在云端数据库中维护。

### 9.2 入站

指定 Incoming 目录中的新文件或新版本按以下过程处理：

1. 使用 ETag、大小、修改时间和路径发现变化。
2. 断点下载到隔离区。
3. 进行病毒扫描、PDF 校验、文件名解析和哈希去重。
4. 通过后创建草稿或待提交记录。
5. 不直接覆盖已批准的图纸版本。

### 9.3 出站

云端审批或 PDM 发布事件触发：

1. 选择应发布的原始、审查或签后 PDF。
2. 使用临时文件名上传。
3. 完成后原子重命名或按 WebDAV 能力进行等价操作。
4. 回读 ETag、大小和哈希进行校验。
5. 记录发布结果和审计事件。

### 9.4 冲突与删除

- 同一路径不同哈希进入冲突队列。
- 管理员可以导入为新版本、保留云端版本或重命名远端文件。
- 默认不传播删除，只记录远端缺失。
- 删除云端权威文件必须经过产品内权限、确认和审计流程。
- WebDAV 凭据存入云密钥服务，不写入数据库明文或日志。
- PostgreSQL 租约或行锁确保同一同步任务只由一个 Worker 执行。

## 10. 数据模型

### 10.1 身份与权限

主要对象：

- users
- invitations
- project_memberships
- roles/capabilities
- sessions
- mfa_factors
- recovery_codes
- login_audit_events

角色定义操作能力，项目成员关系定义数据可见范围。管理员不是默认可跳过所有业务约束；高风险越权操作必须单独声明并审计。

### 10.2 图纸与审批

主要对象：

- documents
- drawing_revisions
- stored_objects
- approval_cases
- review_decisions
- signature_placements
- render_artifacts
- print_archive_events

图纸版本发布后不可原地修改。同一零件和版本保持唯一。审批实例引用明确的图纸版本，不能因后续文件覆盖而改变历史内容。

### 10.3 批注与问题

主要对象：

- annotations
- issues
- issue_comments
- issue_events

annotation 保存页面、图形、标准化坐标、样式和版本号。issue 保存严重级、负责人、期限和状态。状态变化写入不可变事件历史。

### 10.4 PDM

保留并迁移：

- parts
- drawing_revisions 与 PDM 版本关系
- part_usages
- current/effective revision
- approval source links

PDM 不复制文件内容，只引用权威对象和图纸版本。

### 10.5 任务与同步

主要对象：

- background_jobs
- outbox_events
- notifications
- webdav_connections
- webdav_directory_mappings
- webdav_sync_items
- webdav_sync_conflicts
- audit_events

任务具有幂等键、状态、尝试次数、下次执行时间和死信原因。失败必须可查询、可重试、可审计。

## 11. API 与前端数据访问

### 11.1 API v2

新增按领域组织的 `/api/v2`：

- identity
- tasks
- documents
- approvals
- annotations/issues
- parts/revisions
- webdav-sync
- administration

共享 Zod 模型定义请求和响应。统一：

- 错误结构和错误码。
- 分页、筛选和排序。
- 请求 ID。
- 幂等键。
- 乐观并发版本。

旧 API 在迁移期保留受测试的兼容层。每条垂直链路迁移完成后删除对应旧客户端和服务端分支，不长期维护两套真相。

### 11.2 前端领域客户端

替换单体 `api.ts`，按领域拆分：

- identityClient
- taskClient
- documentClient
- approvalClient
- annotationClient
- pdmClient
- syncClient
- adminClient

共享请求层只负责会话、请求 ID、错误解析、取消和重试策略。

## 12. 公网安全

### 12.1 账号

- 禁止开放注册。
- 管理员发送一次性邀请。
- 密码使用适合服务端部署的强哈希算法和独立参数配置。
- 所有账号启用 TOTP MFA，并生成一次性恢复码。
- 管理员可以撤销会话和重置 MFA，但操作必须审计。

### 12.2 会话与请求

- 使用 HttpOnly、Secure、SameSite Cookie。
- 状态修改请求使用 CSRF 防护。
- 会话短期有效并支持轮换、撤销和设备列表。
- 登录、找回密码和 MFA 进行速率限制和锁定保护。
- 服务端逐接口校验角色和项目权限，不依赖前端隐藏按钮。

### 12.3 文件

- 上传先进入隔离区。
- 校验大小、扩展名、MIME、魔数和 PDF 可解析性。
- 执行恶意文件扫描。
- 存储桶私有，下载使用短时签名 URL 或受控代理。
- 不向客户端返回服务器路径、对象存储密钥或内部错误。

### 12.4 审计

必须审计：

- 登录、MFA、邀请和会话撤销。
- 权限和项目成员变化。
- 审批、问题和版本状态变化。
- 文件删除、作废和强制关闭。
- WebDAV 连接、冲突处理和凭据变化。
- 备份、恢复、清理和部署操作。

## 13. 错误处理

### 13.1 原则

- 错误不能被宽泛 `try/catch` 吞掉。
- 前端展示用户可理解的信息和恢复动作。
- API 返回稳定错误码、请求 ID 和可重试标志。
- 日志保留技术细节，但不记录密码、Cookie、Token、PDF 内容或 WebDAV 凭据。
- 后台任务使用有限重试、指数退避和死信状态。

### 13.2 关键场景

- PDF 保存失败：保留本地草稿，显示未保存，允许幂等重试。
- 签章失败：状态保持失败，用户和管理员可查看原因并重试。
- WebDAV 冲突：显示两端版本、哈希和时间，要求人工决策。
- 应用实例故障：负载均衡摘除实例，其他实例继续服务。
- 数据库故障切换：请求短暂失败时有限重试，超过范围则明确提示。
- 对象存储失败：业务状态不提前标记成功，任务保持可恢复。

## 14. 香港云端部署

### 14.1 拓扑

- 香港区域 CDN/WAF 和负载均衡。
- 两个不同可用区的 Linux 应用主机。
- 每台主机使用 Docker Compose 运行 Web/API、Worker 或监控代理。
- 托管 PostgreSQL 多可用区高可用。
- 香港区域私有对象存储，开启版本化和加密。
- 备份存放在独立故障域，并定期验证恢复。

云厂商选择不是本设计的业务约束。实施计划应选择能够在香港区域提供负载均衡、托管 PostgreSQL、高可用对象存储、WAF 和密钥管理的供应商，并保持应用镜像、数据库和 S3 接口可迁移。

### 14.2 发布

- 基础设施通过可审阅的配置管理，避免手工创建不可复现资源。
- CI 构建签名镜像并执行测试、依赖扫描和镜像扫描。
- 数据库使用向后兼容的 expand/contract 迁移。
- 应用逐实例滚动发布，连接排空后再替换下一台。
- 健康检查失败自动停止发布。
- 应用镜像可一键回退。

### 14.3 可用性目标

- 月度 SLO：99.9%。
- 滚动发布无计划停机。
- 单应用实例故障不导致整体不可用。
- RPO 不超过 5 分钟。
- RTO 不超过 30 分钟。

绝对 100% 不停机不作为可验证承诺。以上指标通过故障演练、云服务 SLA 和监控数据验证。

### 阶段 0：基线

- 盘点所有页面、接口、权限、状态、数据库表和文件目录。

### 阶段 1：云与安全底座

- 基础设施、镜像、CI/CD 和可观测性。
- PostgreSQL 与对象存储适配层。
- 邀请制账号、MFA、Cookie 会话和项目权限。

### 阶段 2：应用外壳

- 路由、错误边界、加载和空状态。

### 阶段 4：其余业务链路

- 提交、签名定位和批量提交。

### 阶段 5：迁移演练

- 使用脱敏数据库和文件副本反复执行迁移。

### 阶段 6：正式切换

1. 提前公告 1 至 2 小时维护窗口。

4. 执行增量迁移。
5. 完成数据、文件和角色关键路径门禁。
6. 设置最终 Go/No-Go 点。

- 执行备份恢复和单实例故障演练。
- 修复问题并删除迁移兼容层。

## 16. 迁移与回滚

### 16.1 数据迁移

- 保留稳定主键或维护明确的旧 ID 到新 ID 映射。
- 现有活动账号预创建为邀请账号，在正式切换前完成新密码设置和 TOTP MFA 绑定；旧密码哈希不作为公网长期认证方案。
- 旧系统没有项目成员边界，迁移时先为现有用户保留其原有可见项目范围，再由管理员逐步收紧，避免上线即丢失访问权限。
- 旧 `printer` 角色迁移为明确的打印归档能力；无法映射或长期未使用的账号禁用并写入迁移报告。
- 时间统一存 UTC，界面显示 `Asia/Shanghai`。
- SQLite 文本状态转换为 PostgreSQL 枚举或检查约束前先做未知值扫描。
- 所有外键、唯一约束和当前版本指针在迁移后重建并验证。

### 16.2 文件迁移

- 从旧路径读取文件并计算哈希。
- 按稳定对象键上传对象存储。
- 回读并验证大小和哈希。
- 数据库只在对象验证完成后指向新对象。
- 缺失、无效和重复文件进入迁移异常报告，不静默跳过。

### 16.3 回滚

- 正式开放写入前设置最终 Go/No-Go 点。
- 门禁失败时不开放新系统写入，恢复旧域名和旧系统。
- 应用发布使用旧镜像回退。
- 数据库迁移采用 expand/contract，避免应用回退时遇到破坏性 Schema。
- 新系统开放写入后优先修复前进；若必须回退，先冻结写入并导出增量变化，不能用旧数据库直接覆盖新数据。

## 18. 验收标准

- 现有审批、签名、批注、PDM、归档和运维核心能力无回归。
- 所有角色可从统一任务入口完成主要工作。
- PDF 工作台达到三栏布局、混合批注和问题闭环设计。
- 高严重级问题阻断规则在前后端一致。
- WebDAV 导入和发布可幂等重试，冲突不静默覆盖。
- SQLite 和旧文件迁移通过数据、外键、哈希和可读性门禁。
- 公网只允许邀请账号，MFA、项目权限和审计生效。
- 单应用实例故障和滚动发布不造成整体不可用。
- 备份恢复、RPO/RTO 和月度 SLO 可通过演练和监控验证。
- 桌面和手机真实页面无关键遮挡、溢出和不可访问控件。
- 构建、类型检查、目标测试、浏览器 E2E、性能和安全门禁全部通过。

## 19. 主要风险与控制

### 19.1 范围风险

“整体重构”容易变成无边界的功能扩张。控制方式是按本设计的垂直阶段实施，每阶段只迁移明确业务链路。BOM、工程变更和外部协作另立规格。

### 19.2 数据风险

SQLite 到 PostgreSQL、路径到对象存储的迁移可能暴露历史脏数据。通过脱敏演练、异常报告、哈希校验和 Go/No-Go 门禁控制。

### 19.3 双向同步风险

WebDAV 易形成第二套真相或误删。通过受控目录、云端权威、不可变版本、冲突队列和默认不传播删除控制。

### 19.4 PDF 性能风险

大文件、长页数和多人标注可能阻塞 UI 或 Worker。通过虚拟化、后台任务、限额、队列和压测控制。

### 19.5 高可用复杂度

双实例和托管服务不自动等于高可用。必须验证无状态、连接排空、数据库切换、对象存储恢复和 Worker 租约。

### 19.6 安全风险

公网登录和文件上传扩大攻击面。通过邀请制、全员 MFA、WAF、限流、隔离扫描、私有对象存储、项目权限和审计控制。

## 20. 后续流程

本设计和 UI 设计系统子规格经书面审阅确认后，下一步创建详细实施计划。实施计划应把每个垂直阶段拆成可测试、可审阅、可回退的小任务，并明确影响文件、数据库迁移、测试命令、云资源和发布门禁。
