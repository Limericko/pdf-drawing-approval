- 日期：2026-07-10

## 1. 结论

Phase 1 在不改动现有正式数据、不停止现有局域网系统的前提下，建立云端平台所需的 PostgreSQL、选择式文件存储、邀请制身份体系和 PostgreSQL Worker/Outbox 基础。

本阶段采用“并行平台层”方案：

- `legacy` 运行模式继续使用 SQLite、本地文件和 v1 Bearer JWT，保护现有运行方式。
- `platform` 运行模式使用 PostgreSQL、v2 API、Cookie 会话、项目权限、对象存储抽象和后台 Worker。
- v1 与 v2 不共享写路径，不进行 SQLite/PostgreSQL 或本地文件/S3 双写。
- Phase 1 不迁移正式数据，不创建收费云资源，不执行生产切换。

本地平台环境使用 PostgreSQL、MinIO 和 Mailpit。生产代码保持厂商中立；未来部署到阿里云香港时，分别映射到托管 PostgreSQL、OSS、负载均衡、WAF 和密钥管理服务。

## 2. 背景与目标

最终产品是统一云端运行的工程图纸协同平台。办公室和家庭用户通过同一 HTTPS 域名直接登录，不安装零信任 VPN 客户端，也不保留办公室本地业务服务器。目标容量为 30 至 100 个账号、10 至 30 人并发。

Phase 1 只交付后续垂直重构所需的稳定底座：

- 可在本地重复启动的云端依赖栈。
- PostgreSQL 连接、迁移、事务和 Repository contract。
- filesystem/S3 两种可替换但不可同时启用的存储实现。
- 邀请、密码、强制 TOTP、恢复码、Cookie 会话、CSRF、限流和审计。
- 平台角色能力与项目成员关系。
- PostgreSQL-backed Worker/Outbox、租约、幂等、重试和死信。
- 真实集成测试和浏览器关键路径。

## 3. 范围边界

### 3.1 本阶段包含

- `infra/local` 中的 PostgreSQL、MinIO 和 Mailpit。
- `src/server/platform` 下的平台配置、数据库、存储、任务和安全基础设施。
- `src/server/modules/identity` 下的 v2 身份与项目权限模块。
- 首个管理员启动命令、管理员邀请用户、用户设置 MFA 和登录流程。
- 邀请邮件作为第一条真实 Outbox/Worker 链路。
- filesystem 和 MinIO 共用的存储契约测试。
- `legacy` 与 `platform` 两种运行模式的启动校验和回归保护。

### 3.2 本阶段不包含

- SQLite 与现有文件的正式迁移。
- 图纸、审批、标注、签章、PDM 的 v2 业务迁移。
- WebDAV 双向同步、ClamAV 扫描和 PDF 后台处理。
- 正式阿里云资源、域名、证书、WAF 或生产数据库。
- 多租户、开放注册、社交登录、SSO、短信 MFA。
- 本地与云端双写、离线审批或自动故障降级到旧数据源。

## 4. 运行模式与模块边界

新增显式配置 `PDF_APPROVAL_RUNTIME_MODE=legacy|platform`，默认保持 `legacy`，避免升级后意外改变现有部署。

### 4.1 Legacy 模式

- 沿用当前 `src/server` 启动路径、SQLite、本地目录、v1 路由和 Bearer JWT。
- 保留现有注册与账号行为，Phase 1 不重写旧身份链路。
- 不连接 PostgreSQL、MinIO 或平台 Worker。
- 现有 Electron、服务端 EXE 和局域网流程继续运行。

### 4.2 Platform 模式

- 只挂载 `/api/v2` 平台路由，不暴露 v1 自注册和 JWT 登录。
- 必须连接 PostgreSQL，并选择 `filesystem` 或 `s3` 中的一个权威存储。
- 启动 Web 进程和独立 Worker 进程，共享数据库契约但不共享进程生命周期。
- 使用邀请制身份、Cookie 会话和项目级授权。

### 4.3 代码边界

```text
src/server/
  platform/
    config/       环境变量解析和启动校验
    database/     pg pool、事务、迁移状态
    storage/      StorageAdapter、filesystem、S3
    jobs/         Outbox、Dispatcher、Worker、租约
    security/     会话、CSRF、MFA、限流、审计
  modules/
    identity/     用户、邀请、项目成员、权限
migrations/postgres/
infra/local/
```

业务模块只能通过 Repository、StorageAdapter 和 JobPublisher 使用基础设施。路由不直接执行 SQL，数据库层也不引用 Express。

## 5. PostgreSQL 设计

### 5.1 连接与事务

- 使用 `pg` 连接池，不按请求新建连接。
- Web、Worker 和迁移使用独立且受限的数据库角色；应用角色不是超级用户。
- Web 与 Worker 分别设置有上限的连接池，为迁移和运维保留连接容量。
- 所有查询参数化；事务回调保持短小，不在事务内发送邮件或访问对象存储。
- 设置连接、查询和事务超时，超时作为显式错误暴露。
- 生产可在应用前增加 PgBouncer 或托管连接代理，但业务代码不依赖具体代理。

### 5.2 标识、类型与约束

- 对外实体 ID 使用应用生成的 UUIDv7，兼顾不可枚举和索引局部性。
- 时间使用 `timestamptz`，布尔值使用 `boolean`，哈希与密文使用 `bytea`，无意义长度限制的字符串使用 `text`。
- 状态使用 `text + check constraint`，不使用数据库 enum，降低后续迁移成本。
- 邮箱写入前完成 Unicode 规范化、去除首尾空白和小写化，并由数据库唯一约束兜底。
- 所有外键列建立索引；外键删除行为逐表明确，不依赖隐式级联。
- 高频查询使用与访问模式一致的复合或部分索引，不为尚不存在的查询预建索引。

### 5.3 首批数据表

身份与权限：

- `users`：邮箱、显示名、密码哈希、状态、平台角色、MFA 状态和时间戳。
- `invitations`：令牌哈希、目标邮箱、预设角色、邀请人、过期时间、接受时间。
- `totp_credentials`：加密后的 TOTP 密钥、密钥版本、确认时间。
- `recovery_codes`：单个恢复码哈希、使用时间；每个恢复码只能消费一次。
- `mfa_challenges`：短期挑战令牌哈希、用户、过期时间、尝试次数和完成时间。
- `sessions`：不透明令牌哈希、用户、绝对过期、空闲过期、最近活动、撤销时间和客户端摘要。
- `projects`：用于 Phase 1 权限验证的最小项目记录。
- `project_members`：项目、用户、项目角色和成员状态，项目与用户组合唯一。
- `security_rate_limit_buckets`：只服务公网认证等敏感入口的共享限流窗口。
- `audit_events`：操作者、动作、目标、请求关联 ID、结果和最小必要元数据。

存储与任务：

- `storage_objects`：驱动、对象 key、状态、大小、SHA-256、媒体类型和清理时间。
- `outbox_events`：事务内产生的业务事件、载荷、创建时间和派发时间。
- `jobs`：任务类型、版本化载荷、幂等键、状态、尝试次数、下次执行、租约和错误摘要。
- `schema_migrations`：迁移版本、校验和和执行时间。

活跃会话、未消费邀请、待执行任务和未派发 Outbox 使用部分索引。审计元数据和 Job 载荷保留为受控 `jsonb`，不把稳定业务字段藏入 JSON。

### 5.4 迁移规则

- 迁移是按编号排序的 SQL 文件，由独立命令执行。
- 执行迁移前获取 PostgreSQL advisory lock，防止多实例并发迁移。
- 每个已执行文件保存 SHA-256 校验和；发布后修改历史迁移会被拒绝。
- 单个可事务化迁移在事务内执行；不能事务化的操作必须单独标注并提供恢复步骤。
- Phase 1 迁移只新增平台 Schema，不读取或修改 SQLite。
- 平台服务启动时验证预期 Schema 版本。版本落后或超前都明确拒绝启动，不自动迁移、不静默回退。
- 回退以恢复上一应用镜像并保留兼容的新增 Schema 为主；破坏性回滚留到正式迁移阶段设计。

## 6. 身份与公网安全

### 6.1 账号生命周期

- `platform` 模式关闭公开注册。
- 首个管理员由一次性启动命令创建；命令只在用户表为空时成功，并使用数据库锁防止并发创建。
- 管理员通过邀请创建后续账号。邀请原始令牌只发送给用户，数据库只保存哈希。
- 邀请邮件异步发送时，Worker 根据邀请 ID 和版本化 HMAC 密钥重建原始令牌；Outbox/Job 只保存邀请 ID，不保存原始令牌或其明文副本。
- 邀请默认 24 小时失效、只能使用一次；重新邀请生成新令牌并使旧令牌失效。
- 用户接受邀请后先设置密码并绑定 TOTP，在 TOTP 验证完成前不能建立正式会话。

### 6.2 密码与 MFA

- 密码使用 Argon2id；参数在实现时以当前 OWASP 基线和目标服务器实测为准，并把参数编码在哈希中以支持升级。
- TOTP 对所有平台用户强制启用，允许前后一个时间步的时钟偏差。
- TOTP 密钥使用版本化应用密钥加密保存；生产密钥最终映射到云密钥管理，不能写入仓库或数据库配置表。
- 生成一组一次性恢复码，数据库逐条保存哈希。使用后原子标记，不能重复消费。
- 密码验证成功只产生短期 MFA 挑战；TOTP 或恢复码通过后才创建正式会话。

### 6.3 会话与 CSRF

- 正式会话使用高熵不透明令牌，数据库只保存令牌哈希。
- 生产 Cookie 使用 `HttpOnly`、`Secure`、`SameSite=Lax`、`Path=/`，不设置宽泛 Domain。
- 默认绝对有效期 12 小时、空闲有效期 60 分钟；最近活动时间采用节流更新，避免每个请求都写数据库。
- 登录、MFA 完成、密码修改和权限敏感操作后轮换或撤销相关会话。
- 所有修改请求必须携带与会话绑定的 CSRF Token；前端只保存在内存，通过同源、禁止缓存的 v2 会话接口重新获取。
- 未认证入口要求 JSON Content-Type，并校验 `Origin`/`Sec-Fetch-Site`。平台模式默认不开放跨域凭据访问。

### 6.4 限流、授权与审计

- 登录、MFA、邀请接受和恢复码入口使用 PostgreSQL 共享限流，不能只依赖单进程内存。
- 限流同时考虑来源地址和账号标识；错误响应不泄露邮箱是否存在。
- 平台权限由“平台角色能力 + 项目成员关系”共同决定。管理员也必须通过显式能力执行安全操作。
- Phase 1 不引入 PostgreSQL RLS，以免产生应用授权和数据库授权两套业务真相；权限检查集中在服务层并由 Repository 查询约束兜底。
- 登录、登录失败、MFA、邀请、角色变更、成员变更、会话撤销和安全配置变更写入审计。
- 应用数据库角色只能追加审计记录，不能更新或删除；运维数据库管理员权限仍由部署制度约束。

## 7. 文件存储

### 7.1 选择式后端

统一 `StorageAdapter` 提供流式写入、流式读取、元数据查询和删除：

- `filesystem`：开发、测试和独立部署。
- `s3`：本地 MinIO 和生产阿里云 OSS 的协议实现。

同一环境只允许一个 `PDF_APPROVAL_STORAGE_DRIVER`。配置多个后端、配置不完整或健康检查失败都会明确报错；系统不双写，也不从失败的 S3 隐藏降级到本地目录。

### 7.2 对象规则

- 对象 key 由服务端生成，包含受控前缀和不可变 ID，不使用用户文件名拼接路径。
- filesystem 实现必须解析规范路径并证明目标仍位于配置根目录内，拒绝路径穿越和符号链接逃逸。
- Bucket 保持私有，服务端凭据按最小权限配置；Phase 1 不提供公共对象 URL。
- 文件写入时流式计算 SHA-256 和字节数，避免把完整 PDF 读入内存。
- 对象不可原地覆盖；新版本产生新 key，保证缓存、审计和恢复语义稳定。

### 7.3 写入与删除一致性

写入流程：

1. PostgreSQL 创建 `staging` 存储记录和系统对象 key。
2. 在数据库事务外流式写入当前权威存储，并计算大小和 SHA-256。
3. 写入完成后通过 `head` 校验对象元数据。
4. 短事务把记录改为 `ready`；只有 `ready` 对象能被业务层读取。
5. 失败记录保持可诊断状态，由清理任务回收未完成或孤立对象。

数据库是对象元数据和业务引用的权威，存储后端是字节内容的权威。两者无法组成分布式事务，因此采用不可变 key、显式状态和可重入清理实现最终收敛，不宣称不存在失败窗口。

删除先解除业务引用并标记待清理，再由 Worker 物理删除。删除成功后记录清理时间；暂时失败进入重试，避免数据库回滚后文件已经不可恢复。

## 8. Worker 与 Outbox

### 8.1 数据流

需要外部副作用的业务事务同时写入业务状态与 `outbox_events`。Dispatcher 领取未派发事件，并在同一短事务中：

1. 使用 `outbox:<event-id>:<handler-version>` 生成唯一幂等键。
2. 插入或确认已存在的 Job。
3. 标记 Outbox 已派发。

首条真实链路是创建邀请后发送邀请邮件。本地由 Mailpit 接收，生产邮件提供方留到部署阶段映射。

邀请链接把令牌放在 URL fragment 中，由浏览器通过 JSON body 提交，避免令牌进入代理路径、访问日志或 Referer。

### 8.2 领取、租约和重试

- Worker 使用 `FOR UPDATE SKIP LOCKED` 原子领取可执行 Job，多 Worker 不互相等待。
- 领取事务只更新状态、Worker ID 和租约；SMTP、S3 等外部调用在提交后执行。
- 长任务可续租；进程崩溃后，过期租约由其他 Worker 回收。
- 只对超时、连接中断和明确的临时依赖错误重试。
- 校验失败、权限失败和不支持的任务版本直接进入失败或死信。
- 重试使用带抖动的指数退避和有限次数；超过上限进入死信，禁止无限循环。

系统提供“至少一次”处理语义。内部处理器必须使用幂等键和条件更新消除重复；对不能提供事务幂等的外部系统，只能使用稳定消息 ID、可重入设计和审计降低重复副作用，不能承诺绝对恰好一次。

Phase 1 不开发任务管理 UI，只提供结构化日志、队列深度、最老任务年龄、Worker 心跳、死信计数和受控诊断/重试命令。

## 9. 错误、日志与运行状态

### 9.1 API 错误

- v2 API 使用 `application/problem+json`。
- 返回稳定的机器错误码、HTTP 状态、用户可理解标题和请求关联 ID。
- 校验、认证、授权、冲突、限流、依赖不可用和内部错误明确区分。
- 响应不包含堆栈、SQL、对象 key、哈希、密钥或账号存在性信息。

### 9.2 日志与关联

- 每个请求生成或接受受信格式的关联 ID，并传播到审计、Outbox 和 Job。
- 结构化日志包含模式、模块、操作、耗时和结果，不记录 Cookie、密码、邀请令牌、TOTP、恢复码或存储凭据。
- 可预期的业务拒绝使用合适日志级别，真实异常保留堆栈供服务端诊断。

### 9.3 健康检查

- `/health/live` 只证明进程事件循环仍可响应。
- `/health/ready` 验证 PostgreSQL、预期迁移版本和当前存储后端。
- Worker 心跳、队列积压和邮件依赖单独报告；邮件短暂失败不应让核心 API 被负载均衡摘除。
- 检查使用短超时并缓存高频依赖结果，避免健康检查本身放大故障。

## 10. 配置与启动校验

平台配置由单一解析器读取并返回不可变、已校验的配置对象。至少包括：

- 运行模式、服务地址和可信代理设置。
- PostgreSQL URL、连接池上限和超时。
- `filesystem|s3` 驱动及对应根目录或 endpoint、region、bucket 和凭据。
- Cookie、会话、CSRF、Argon2、TOTP 加密密钥和限流参数。
- SMTP/Mailpit 地址和 Worker 并发、租约、重试参数。

规则如下：

- `legacy` 忽略未使用的平台依赖，但继续校验现有关键配置。
- `platform` 缺少任何必要配置都拒绝启动。
- 生产环境拒绝默认密钥、弱 Cookie 配置、公开 Bucket 或开发邮件设置。
- 密钥只从环境变量或部署密钥挂载读取，不写入日志、示例值或前端产物。
- 数据库迁移是显式发布步骤，Web 与 Worker 启动时只校验版本。

## 11. 本地依赖栈与阿里云映射

`infra/local` 提供固定版本镜像、健康检查和命名卷：

- PostgreSQL：平台数据与队列。
- MinIO：S3 contract 和对象存储集成测试。
- Mailpit：邀请邮件捕获和浏览器验收。

WebDAV 和 ClamAV 在 Phase 1 不启动，避免提前引入未使用的运行成本。

代码层只依赖 PostgreSQL、S3 协议和 SMTP 契约。未来阿里云香港映射不改变业务接口：

- 应用容器映射到 ECS 上的容器运行环境。
- PostgreSQL 映射到香港区域的托管高可用数据库。
- S3 实现映射到私有 OSS Bucket。
- TLS、WAF、负载均衡、密钥管理、备份和跨可用区部署在 Phase 6 落地。

## 12. 测试策略

### 12.1 单元与契约

- 配置解析、密码策略、令牌哈希、权限矩阵、错误映射和重试分类单元测试。
- 每个 Repository 使用同一 contract 在真实 PostgreSQL 上执行。
- filesystem 与 MinIO 使用同一 StorageAdapter contract，覆盖流式写入、校验、缺失对象、路径安全和清理。
- 后端测试拆成可独立执行的分组，每组设置 60 秒硬超时。

### 12.2 身份集成

覆盖：

- 空库首管理员创建及非空库拒绝。
- 邀请过期、重复使用、撤销和并发接受。
- Argon2 密码验证、强制 TOTP、恢复码单次消费和挑战过期。
- Cookie 属性、会话轮换、绝对/空闲过期、撤销和多实例共享。
- 缺失、错误或跨会话 CSRF Token。
- 登录、MFA、邀请等入口的共享限流。
- 平台能力与项目成员关系的允许/拒绝矩阵。
- 安全操作审计完整性和敏感字段脱敏。

### 12.3 Worker 与故障恢复

覆盖：

- 多 Worker 并发领取而不重复认领。
- Outbox 派发幂等、Worker 崩溃后的租约回收。
- 临时错误退避、永久错误不重试、达到上限进入死信。
- 邀请邮件在 Mailpit 可见且不包含日志泄漏。
- 存储写入在数据库失败、对象写入失败和进程中断后的收敛。

### 12.4 浏览器与兼容回归

浏览器关键路径完成：首管理员登录、邀请用户、接受邀请、设置 TOTP、登录、读取 CSRF、访问允许项目并拒绝未授权项目。

同时执行 Phase 0 已建立的 legacy 客户端、服务端、Electron 和 Playwright 基线，证明 v1、SQLite 和本地文件工作方式未受影响。

## 13. 验收标准

Phase 1 完成必须同时满足：

1. 本地依赖栈可通过单一文档化命令启动、检查和停止。
2. 新库能显式迁移到目标版本，重复运行无额外变化，篡改历史迁移会失败。
3. PostgreSQL Repository contract 全部通过，连接池和事务在异常后可恢复。
4. filesystem 与 MinIO 的 StorageAdapter contract 完全一致。
5. 身份、安全、权限和审计集成测试全部通过。
6. Worker 并发、租约、幂等、退避和死信测试全部通过。
7. 浏览器完成完整邀请、MFA、登录和项目授权流程。
8. `legacy` 模式全部 Phase 0 门禁继续通过。
9. `platform` 配置错误明确拒绝启动，不出现隐藏降级或第二套数据真相。
10. 正式数据、现有生产目录和收费云资源均未被访问或修改。

## 14. 风险与控制

- **两套运行模式长期共存：** Phase 1 只为迁移期保留模式开关；Phase 4 完成业务迁移后必须提供 v1 零引用证明并删除 legacy 分支。
- **数据库和对象存储无法原子提交：** 使用不可变对象、显式状态、哈希校验和可重入清理，不使用危险双写补偿。
- **公网身份攻击：** 关闭注册、强制 MFA、共享限流、同源校验、CSRF、短挑战和安全审计共同防护。
- **任务重复执行：** 明确至少一次语义，使用幂等键、条件更新和稳定外部消息 ID。
- **厂商绑定：** Phase 1 只依赖 PostgreSQL、S3 和 SMTP 契约；阿里云专属资源只出现在部署映射与 Phase 6 配置中。
- **Phase 1 范围膨胀：** 不提前实现图纸、审批、WebDAV、扫描或生产部署，只建立后续垂直链路必需的底座。

## 15. 已确认决策记录

- 云端统一部署，办公室和家庭用户通过公网 HTTPS 登录。
- 云区域为中国香港；不影响仓库内的厂商中立设计。
- 采用分阶段垂直重构和并行平台层。
- 平台模式使用 PostgreSQL，旧系统在 legacy 模式继续使用 SQLite。
- 存储采用选择式后端：filesystem 或 S3，同一环境只允许一个权威存储。
- 公网平台关闭开放注册，强制 TOTP MFA，使用 Cookie 会话和 CSRF。
- WebDAV 双向同步保留在 Phase 5，本阶段不实现。
- Phase 1 不迁移正式数据，不影响现有局域网运行方式。
