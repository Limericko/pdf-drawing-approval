# Windows 局域网部署说明

## 1. 准备审批服务器

选择一台办公室内常开的 Windows 电脑作为审批服务器。

建议：

- 设置固定局域网 IP，或使用固定电脑名。
- 安装坚果云客户端。
- 同步 `图纸审批` 根目录。
- 若使用“免 Node 服务端 exe”，不需要安装 Node.js。
- 若使用源码服务包或开发模式，安装 Node.js LTS。

## 2. 安装依赖

在项目目录执行：

```powershell
npm install
```

## 3. 启动服务

开发或试运行：

```powershell
npm run dev
```

或者使用脚本：

```powershell
.\scripts\start-server.ps1
```

默认访问地址：

```text
http://审批电脑IP:8080
```

示例：

```text
http://192.168.1.20:8080
```

## 4. 初始账号

首次启动会自动创建：

```text
管理员：admin / admin123
主管：supervisor / 123456
工艺：process / 123456
```

打印归档由设计师在审批详情页执行，不再需要单独打印账号。上线前应修改默认密码，并按团队实际人员补充设计师账号。

## 5. 配置坚果云目录

用管理员登录后，在“配置”页设置：

```text
watch_root = 坚果云本地同步的图纸审批根目录
```

示例：

```text
D:\Nutstore\图纸审批
```

目录结构应为：

```text
图纸审批/
  01-待提交/
  02-审批中/
  03-已驳回/
  04-已通过待打印/
  05-已打印归档/
```

修改监听目录后，重启服务生效。

## 6. 配置邮件通知

在“配置”页设置：

```text
app_base_url = http://审批电脑IP:8080
smtp_host = SMTP服务器
smtp_port = 465 或 587
smtp_user = 邮箱账号
smtp_password = 邮箱授权码或密码
smtp_from = 发件人地址
supervisor_email = 主管邮箱
process_email = 工艺邮箱
```

填写后可在“系统管理 -> 目录与通知”中使用“发送测试邮件”验证 SMTP 配置。

测试失败时，页面会显示错误信息，并在“操作日志”中记录失败原因。

## 7. 第四版上线检查

第四版正式试运行前，管理员先完成以下检查：

1. 修改默认账号密码。
2. 在“系统管理 -> 目录与通知”配置审批根目录。
3. 点击“创建标准目录”，确认五个标准目录都存在。
4. 点击“重启服务”，让监听目录生效。
5. 在“系统管理 -> 用户管理”补齐设计师、主管、工艺和管理员账号。
6. 设计师、主管、工艺分别进入“我的签名”，上传或手写保存签名。
7. 在“系统管理 -> 签名模板”确认至少有一个常用图框模板，或先在审批详情页保存一个模板。
8. 在“系统管理 -> 运维追溯”确认：
   - 风险看板没有需要立即处理的异常。
   - 系统健康诊断不是异常。
   - 标准目录为 `5/5`。
   - 写入权限为 `5/5`。
   - 关键签名用户已配置。
9. 点击“立即备份”，确认备份记录生成。
10. 用一张真实图纸完成单张提交、并行审核、自动签名、打印归档。
11. 用多张真实图纸完成一次批量上传，确认每张图纸签名框位置独立保存。
12. 对已通过待打印的图纸试跑一次“批量重新生成签后 PDF”。

## 8. 网页提交图纸与批量上传

第四版推荐设计师通过网页提交。单张和多张 PDF 都走同一个“提交图纸”入口：

1. 设计师登录系统。
2. 进入“提交图纸”。
3. 上传一张或多张 `零件名-a0A0.pdf`。
4. 确认项目名、零件名、版本。
5. 如页面提示“同零件已有版本”，先确认本次版本号是否正确。
6. 可选择签名模板作为初始位置。
7. 在 PDF 预览上拖动和缩放三个签名框：
   - 设计
   - 主管
   - 工艺
8. 如果是批量上传，逐张切换 PDF 并检查签名框位置。
9. 提交审批。

提交成功后，系统会把文件写入：

```text
图纸审批\02-审批中\项目名\
```

审批单进入主管和工艺并行审核。

批量上传规则：

- 每张 PDF 都有自己的签名框位置，不能只检查第一张。
- “批量套用模板”只用于初始化位置，仍应逐张确认。
- “套用到当前图纸”只影响当前选中的 PDF。
- 单个文件失败不会阻塞其它文件，页面会显示逐项结果。
- 文件名、PDF 文件头、重复版本或缺签名框都会导致对应项失败。

## 9. 目录提交图纸

设计师把 PDF 放入：

```text
图纸审批\01-待提交\项目名\零件名-a0A0.pdf
```

系统会在文件稳定后自动移动到：

```text
图纸审批\02-审批中\项目名\
```

如果文件扩展名是 `.pdf`，但文件内容不是有效 PDF，系统会记录为 `PDF 无效`，不会进入审核待办。管理员可在图纸详情页替换 PDF 或重新校验。

目录提交仍作为兼容和兜底方式保留。需要完整自动签名闭环时，优先使用网页提交，因为网页提交会在提交时保存三个签名框。

如果采用目录提交，审批详情页会显示签名位置状态。设计师或管理员可打开详情页，点击“编辑签名框”，在 PDF 预览上放置设计、主管、工艺三个签名框并保存。若主管和工艺已经通过，保存后系统会立即尝试生成签后 PDF。

## 10. 配置签名

每个需要盖章的用户都要自己配置签名：

1. 登录系统。
2. 进入“我的签名”。
3. 上传 PNG 签名，或使用页面手写板绘制签名。
4. 保存后刷新页面，确认能看到当前签名预览。

签名建议：

- 使用深色笔迹。
- 图片尽量裁剪到签名主体。
- PNG 透明背景或白底均可。
- 不要上传扫描件中的整页图片。

管理员可在“系统管理 -> 运维追溯 -> 签名配置”查看设计师、主管、工艺是否已配置签名。

### 签名模板

签名模板用于复用常见图框签字栏位置，减少每次提交都重新定位的工作量。

创建方式：

- 在审批详情页调整好设计、主管、工艺三个签名框后，点击“保存为模板”。
- 管理员进入“系统管理 -> 签名模板”，新建或维护模板。

使用方式：

- 提交图纸时选择模板，可把模板位置复制到当前图纸。
- 批量上传时可“批量套用模板”初始化所有图纸，也可只套用到当前图纸。
- 模板可设置为全局模板，也可填写适用项目。

注意：

- 套用模板只是复制位置，不会把审批单和模板持续绑定。
- 模板后续修改不会自动改动已提交图纸。
- 批量上传时每张图纸位置不同，套用模板后仍要逐张检查。

## 11. 审批与自动签名

主管和工艺都通过后，系统会自动生成签后 PDF。

生成成功后：

- 审批详情页显示“签名已生成”。
- 审批详情页提供“打开签后 PDF”。
- 签后文件路径写入数据库。
- 签后文件哈希写入数据库。
- 操作日志记录签名生成。

生成失败时：

- 审批详情页显示失败原因。
- 管理员可补齐签名或修复文件后点击重试。
- 如果失败原因是缺少签名框位置，设计师或管理员可在详情页编辑并保存签名位置。
- 操作日志记录失败原因。

常见失败原因：

- 设计师、主管或工艺未配置签名。
- 原始 PDF 被删除或还未同步完成。
- 输出目录不可写。
- PDF 文件内容无效。

## 11.1 V6.1 图纸批注与审查版 PDF

V6.1 增强了系统内图纸批注能力，批注只用于审核沟通和审查版 PDF，不写入正式签后 PDF。

审核人处理图纸时：

1. 在审批详情页左侧 PDF 上选择批注工具。
2. 支持定位、箭头、矩形、圆形、文字、画笔和云线。
3. 先在图纸上标记位置，再填写批注内容并保存。
4. 选中已有批注后可移动、缩放和调整颜色。
5. 右侧批注列表点击任一条，会自动定位到 PDF 对应标记。

设计师处理问题时：

- 可查看所有批注。
- 可点击批注列表定位到图纸位置。
- 可把已处理的问题标记为完成。
- 不能新增审核批注。

输出规则：

- “审查版 PDF”会带上图纸批注，适合问题归档和复核。
- “签后 PDF”只包含正式签名，不包含审核批注。
- 已归档或作废图纸的批注仅可查看，不再允许编辑。

## 12. 打印归档与批量处理

设计师或管理员在审批详情页处理待打印记录。

规则：

- 有签后 PDF 时，优先打开签后 PDF。
- 签名失败或尚未生成时，页面会提示异常，避免误打印未签名文件。
- 标记打印后，审批进入已打印归档状态。

第四版在“全部图纸”页增加批量处理：

- 设计师和管理员可勾选多张 `已通过待打印` 图纸。
- 点击“批量重新生成签后 PDF”，可按最新签名框位置和当前签名资产重新生成签后文件。
- 点击“批量标记打印归档”，可把已生成签后 PDF 的图纸批量归档。
- 批量操作按图纸逐项返回成功或失败，不会因为单张失败中断整批。
- 主管和工艺账号不能执行批量签后 PDF 处理或批量归档。

## 13. 开机自启

可选执行：

```powershell
.\scripts\install-startup-task.ps1
```

这会创建一个 Windows 登录后启动的计划任务。

## 14. 异常处理

管理员可在“全部图纸”中筛选异常状态：

- `PDF 无效`：替换 PDF、重新校验或作废。
- `文件丢失`：重新绑定服务器上的有效 PDF 或作废。
- `文件名异常`：作废后让设计师按标准命名重新提交。
- `已作废`：保留记录，不再参与审批。

重新绑定文件要求：

- 文件存在。
- 文件在当前审批根目录内。
- 文件内容以 `%PDF-` 开头。

签名异常处理：

- `签名失败`：先查看失败原因。
- 缺签名：让对应用户进入“我的签名”配置。
- 缺文件或 PDF 无效：按文件异常流程重新绑定或重新校验。
- 修复后由管理员在详情页点击“重新生成签后 PDF”。

## 15. 手动扫描

如果坚果云同步期间出现漏检，或服务离线期间文件被移动/删除，可在“系统管理 -> 目录与通知”点击：

```text
立即重新扫描
```

扫描会：

- 补偿漏掉的新 PDF。
- 标记丢失的 pending 审批文件。
- 统计处理数、丢失数、无效 PDF 数。
- 在操作日志和扫描记录中留痕。

## 16. 运维追溯

系统管理页的“运维追溯”包含：

- 风险看板。
- 系统健康诊断。
- 数据库备份。
- 签名配置状态。
- CSV 追溯报表导出。
- 全局操作日志。

风险看板应每天先看，重点处理：

- 审批根目录未配置或不存在。
- 标准目录缺失或不可写。
- 最近备份过期。
- 文件丢失或 PDF 无效审批单。
- 签名失败审批单。
- 缺少设计、主管、工艺关键签名。

批量提交记录用于检查网页批量上传是否逐项成功。管理员可查看：

- 批量提交项目名、批次号和提交时间。
- 成功数、失败数和总数。
- 每张 PDF 的处理状态、签名框来源和失败原因。

健康诊断应重点查看：

- 数据库是否可读写。
- 审批根目录是否存在。
- 标准目录是否齐全。
- 标准目录是否可写。
- 最近扫描和最近备份是否正常。

CSV 报表用于归档追溯，可按项目、状态和日期导出。第四版报表包含“同零件版本数”，用于快速发现同项目、同零件的版本链。

## 17. 操作日志

系统管理页提供：

- `运维追溯`：记录用户、审批、扫描、SMTP 测试、备份和重启请求。
- `服务日志`：读取 `server.log` 和 `server.err.log` 尾部内容。

图纸详情页也会显示当前审批单的操作时间线。

## 18. 数据备份

默认数据库位置：

```text
data\pdf-approval.sqlite
```

手动备份命令：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\backup-database.ps1
```

脚本会复制：

```text
data\pdf-approval.sqlite
data\pdf-approval.sqlite-wal
data\pdf-approval.sqlite-shm
```

备份输出目录：

```text
backups\pdf-approval-yyyyMMdd-HHmmss
```

建议每天备份一次，或者在升级系统前手动执行一次。

第三版及第四版都支持在管理端点击“立即备份”。该功能会生成同样格式的备份目录，并在 `backup_runs` 中记录备份结果。

## 19. 恢复备份

恢复前先停止审批服务。

建议步骤：

1. 停止服务。
2. 复制当前 `data` 目录到临时位置，避免二次损坏。
3. 从目标备份目录复制以下文件到 `data`：

```text
pdf-approval.sqlite
pdf-approval.sqlite-wal
pdf-approval.sqlite-shm
```

如果备份目录中没有 `-wal` 或 `-shm` 文件，只恢复存在的文件。

4. 启动服务。
5. 打开 `/health` 确认服务正常。
6. 登录系统检查最近审批单、操作日志和运维诊断。

## 20. 回滚方式

如果系统异常：

1. 停止审批服务。
2. 暂时让设计师和审核人按坚果云文件夹人工流转。
3. 从 `backups` 或其他备份恢复 SQLite 数据库。
4. 检查 `图纸审批` 下各状态目录中的文件位置。

## 21. 常见问题

### 文件夹选择窗口不弹出

Windows 后台服务启动系统弹窗不稳定。优先使用“浏览服务器目录”选择审批根目录。

### 新 PDF 没有进入审批

处理顺序：

1. 确认文件是有效 PDF。
2. 确认文件名符合 `零件名-a0A0.pdf`。
3. 确认服务已重启并监听当前根目录。
4. 在“目录与通知”点击“立即重新扫描”。
5. 查看“服务日志”和“操作日志”。

### 签后 PDF 没有生成

处理顺序：

1. 打开审批详情页查看签名状态和错误信息。
2. 确认设计师、主管、工艺都已配置签名。
3. 如果提示缺少签名框位置，由设计师或管理员在详情页保存设计、主管、工艺三个签名框。
4. 确认原始 PDF 仍存在且有效。
5. 确认 `04-已通过待打印` 可写。
6. 管理员点击“重新生成签后 PDF”。

### PDF 预览加载错误

如果页面提示 PDF 无效，通常是坚果云同步未完成、文件损坏或导出的文件并非真正 PDF。重新导出 PDF 后由管理员重新绑定或重新校验。

## 22. V5 Electron 客户端

V5 当前主线改为 Electron 客户端 + 局域网服务端。Tauri 托盘助手保留为历史实验，不再作为正式客户端实施路径。

服务端仍部署在办公室内一台常开的 Windows 电脑上，继续负责：

- SQLite 数据库。
- 坚果云审批根目录监听。
- PDF 上传、审核、签名、打印归档。
- 用户、权限、日志、备份。

团队电脑安装 Electron 客户端。客户端首次启动填写审批服务器地址，例如：

```text
http://192.168.1.20:8080
```

开发验证命令：

```powershell
npm install --registry=https://registry.npmmirror.com
npm run build
npm run desktop:test
npm run desktop:dev
```

说明：

- `npm run desktop:dev` 会打开 Electron 客户端窗口。
- 客户端会加载本项目构建出的 `dist/client` 前端。
- 客户端本机保存服务器地址，不保存密码。
- 浏览器访问 `http://服务器IP:8080` 仍作为备用入口。
- Electron 不需要 Rust、Cargo、Visual Studio C++ Build Tools 或 WebView2 额外打包链路。

推荐生成 Windows 安装包：

```powershell
npm run installer:package
```

输出：

```text
dist\installers\client\PDF图纸审批客户端-安装包-0.9.1.exe
dist\installers\server\PDF图纸审批服务端-安装包-0.9.1.exe
```

安装方式：

- 服务器电脑安装 `PDF图纸审批服务端-安装包-0.9.1.exe`。
- 设计师、主管、工艺和管理员电脑安装 `PDF图纸审批客户端-安装包-0.9.1.exe`。
- 安装包会创建桌面快捷方式和开始菜单快捷方式。
- 当前安装包未配置企业代码签名，Windows 可能显示安全提醒。确认来源是公司内部构建产物后继续安装。

局域网更新清单：

`npm run installer:package` 会额外生成：

```text
dist\updates\latest.json
dist\updates\latest.yml
dist\updates\CHANGELOG.md
dist\updates\PDF图纸审批客户端-安装包-0.9.1.exe
dist\updates\PDF图纸审批客户端-安装包-0.9.1.exe.blockmap
```

服务端会直接托管安装目录下的 `releases` 发布目录。发布新版时，将文件放成：

```text
PDF图纸审批服务端安装目录\
  releases\
    updates\
      latest.json
      latest.yml
      CHANGELOG.md
      PDF图纸审批客户端-安装包-0.9.1.exe
      PDF图纸审批客户端-安装包-0.9.1.exe.blockmap
    installers\
      client\
        PDF图纸审批客户端-安装包-0.9.1.exe
      server\
        PDF图纸审批服务端-安装包-0.9.1.exe
```

服务端会自动通过当前访问地址提供：

```text
http://服务器IP:8080/updates/latest.json
http://服务器IP:8080/updates/latest.yml
```

管理端不再需要填写更新清单地址。`latest.json` 继续用于网页端版本更新说明和安装包下载地址，`latest.yml` 用于 Electron 客户端启动时自动检查更新。客户端发现新版后会自动下载并显示进度，下载完成后弹出安装步骤，由用户打开安装包按 Windows 向导完成升级，不会静默安装。服务端安装包仍只在管理员运维入口展示。服务端升级仍建议先备份数据库，再运行新版服务端安装包。

从 `0.8.7` 开始，服务端安装包升级时会保留安装目录里的 `data`、`backups`、`logs`、`releases` 和 `server-config.json`。如果旧版本升级后出现 `/updates/latest.json` 返回 404，说明 `releases\updates\latest.json` 缺失，需要重新同步发布目录。

特殊部署如果要使用独立更新服务器，可在服务端启动环境中设置 `PDF_APPROVAL_UPDATE_MANIFEST_URL` 覆盖默认清单地址；常规局域网部署不要设置。

便携包仍作为备用方式保留。

生成便携客户端：

```powershell
npm run desktop:package
```

输出：

```text
dist\desktop-client\PDF图纸审批客户端
```

把整个目录复制到团队电脑，双击 `PDF图纸审批客户端.exe` 启动。

生成服务端发布包：

推荐新手使用免 Node 服务端 exe：

```powershell
npm run server:exe
```

输出：

```text
dist\server-exe\PDF图纸审批服务端
```

把整个目录复制到审批服务器电脑，双击：

```text
PDF图纸审批服务端.exe
```

服务端窗口会显示运行状态、本机地址、局域网地址、启动设置、数据目录、备份目录和日志目录。保持窗口打开，其他电脑访问窗口中的局域网地址。该包已经包含 Electron/Node 运行时，不需要目标电脑安装 Node.js 或执行 `npm install`。

服务端端口：

- 默认端口是 `8080`。
- 在服务端窗口右侧“启动设置”里可修改 HTTP 端口。
- 点击“保存并重启”后，新端口生效。
- 端口配置保存到服务端目录下的 `server-config.json`。
- 如果启动前设置了环境变量 `PORT`，会优先使用环境变量，并覆盖窗口配置。

仍可生成源码服务端发布包：

```powershell
npm run server:package
```

输出：

```text
dist\server-package\PDF图纸审批服务端
```

把整个目录复制到审批服务器电脑后，在该目录执行：

```powershell
npm install --omit=dev --registry=https://registry.npmmirror.com
powershell -ExecutionPolicy Bypass -File scripts\start-server.ps1
```

该包不复制 `node_modules`，并且 `package.json` 只保留服务端运行依赖。

服务端 exe 和源码服务包的运行数据都放在发布目录内：

```text
data      数据库和上传文件
backups   数据库备份
logs      server.log / server.err.log
server-config.json  服务端窗口保存的端口配置
```

区别：

- `server:exe`：适合新手部署，复制整个目录后双击 exe。
- `server:package`：适合管理员或开发维护，目标电脑需要 Node.js 和一次依赖安装。

常见问题：

- 首次打开提示连接服务器：填写审批服务器的局域网地址。
- 提示无法连接：检查服务器是否启动、防火墙是否放行 8080、IP 是否正确。
- 更换服务器 IP：在登录页“审批服务器”处修改并保存，然后重新登录。
- 客户端打不开前端：先在项目根目录执行 `npm run build`。

## 23. 历史 Tauri 托盘助手说明

此前 V5 尝试过 Tauri 托盘助手。该方案用于本机提醒和快捷入口，不替代 Web 审批系统；当前已不作为正式主线。

当前审批服务器/开发机不安装 Tauri 打包环境，只做只读检查和前端验证：

```powershell
.\scripts\check-tauri-prereqs.ps1
.\scripts\build-tray-frontend-only.ps1
```

`check-tauri-prereqs.ps1` 不创建目录、不下载依赖、不安装 Rust 或 Visual Studio。`build-tray-frontend-only.ps1` 只执行托盘单元测试和前端构建。

Windows 安装包由外部构建机、CI、临时虚拟机或已准备好的专用打包电脑产出。外部构建机需要：

- Node.js / npm。
- Rust stable。
- Rust target `x86_64-pc-windows-msvc`。
- Microsoft C++ Build Tools，包含 C++ 桌面开发工具链。
- Microsoft Edge WebView2 Runtime。
- 可访问 npm registry 和 Rust crates 源，或配置可用的公司内部镜像。

托盘助手相关文档：

- 用户使用说明：`docs/tray-helper-user-guide.md`
- 管理员部署说明：`docs/tray-helper-admin-guide.md`
- 外部构建说明：`docs/tray-helper-external-build.md`
- 发布验收清单：`docs/tray-helper-verification.md`

当前机器常用验证命令：

```powershell
npm test
npm run build
.\scripts\build-tray-frontend-only.ps1
```

外部构建机安装包构建命令：

```powershell
npm install --registry=https://registry.npmmirror.com
npm --prefix apps/tray-helper install --registry=https://registry.npmmirror.com
npm test
npm run build
npm run tray:test
npm --prefix apps/tray-helper run build
npm run tray:build
```

构建成功后，安装包应位于：

```text
apps\tray-helper\src-tauri\target\release\bundle
```

当前 Tauri 配置使用产品名 `PDF 图纸审批托盘助手`，应用标识 `local.pdf-approval.tray-helper`，Windows 打包目标为 `nsis` 和 `msi`。托盘图标和安装包图标位于：

```text
apps\tray-helper\src-tauri\icons
```

## 24. V7 连接诊断与自动维护

V7 增强了局域网部署后的排障和长期运行能力。

服务端窗口会显示真实局域网地址，并提供“复制客户端地址”。给同事配置客户端时，优先复制该地址，例如：

```text
http://192.168.1.20:8080
```

不要把 `http://127.0.0.1:8080` 发给其他电脑。`127.0.0.1` 只代表当前电脑，同事电脑填写后会连接到自己本机。

客户端登录页和首次连接页支持“连接自检”，会检查：

- 服务器是否可达。
- 服务端版本。
- API 兼容版本。
- 当前地址是否是 `127.0.0.1`、`localhost` 或格式异常。

管理员在“系统管理 -> 运维追溯”中可配置自动维护：

- 自动数据库备份：建议每天凌晨执行。
- 自动清理维护：清理临时上传、旧失败批量提交记录和未引用的旧签后 PDF。
- 维护执行结果：直接查看自动备份、自动清理、备份校验的最近一次结果。
- 校验备份目录：恢复前可先确认备份目录是否可读、是否包含 SQLite 数据库文件。

PDF 批注和签名定位工作区提供横向缩略页导航，适合多页图纸快速跳页；缩略页只作为导航控件，不重新渲染完整 PDF 页面，避免影响预览性能。

`/health` 会返回 `lanUrls`，开发服务和正式服务都用于展示可给同事填写的局域网访问地址；测试环境保持空列表，避免自动化测试受电脑网卡状态影响。

V7 不改变审批流程：仍然是设计师提交，主管和工艺并行审核，通过后自动生成签后 PDF，设计师执行打印归档。
