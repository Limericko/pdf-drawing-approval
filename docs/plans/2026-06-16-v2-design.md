# PDF 图纸审批系统第二版方案

日期：2026-06-16

## 版本定位

第二版目标是把第一版从“可运行的审批系统”升级为“可日常运营、可纠错、可追溯”的局域网内部工具。

第一版已经完成文件监听、并行审核、目录流转、PDF 预览、系统管理、用户管理和基础通知。第二版不优先扩展复杂协作能力，而是补齐真实使用中最容易卡住的环节：

- 文件异常后如何恢复
- 审批单错误后如何作废
- 管理员如何排查和修复
- 每个关键动作是否有记录
- 邮件和部署是否可靠

## 设计原则

- 先增强稳定性和可维护性，再扩展复杂功能。
- 管理员应能在页面上处理常见异常，不需要直接改数据库。
- 所有关键操作要留下审计记录。
- 文件系统仍以本地坚果云同步目录为核心，不在第二版引入坚果云 API。
- 保持小团队固定审核流程，不做复杂审批流引擎。

## 第二版范围

### 必做

1. 异常审批单处理
   - `file_missing` 文件丢失可重新绑定文件、重新提交或作废。
   - `INVALID_PDF_FILE` 可替换为有效 PDF。
   - `filename_invalid` 可作废或重新扫描。
   - 重复版本提交时给出明确处理结果。

2. 审批单管理增强
   - 新增作废状态。
   - 管理员可作废异常或错误审批单。
   - 全部图纸支持异常状态筛选。
   - 图纸详情显示更清晰的异常处理入口。

3. 操作审计日志
   - 新增操作日志表。
   - 记录审批创建、文件移动、审核、驳回、归档、作废、重新绑定、重新扫描、用户管理、服务重启等动作。
   - 图纸详情显示该图纸的操作时间线。
   - 系统管理页显示全局操作日志。

4. 文件扫描与修复
   - 管理端增加“立即重新扫描”按钮。
   - 显示最近扫描时间、扫描结果和处理数量。
   - 支持针对单个异常审批单重新绑定服务器上的 PDF 文件。

5. 邮件与通知可靠性
   - 管理端增加 SMTP 测试按钮。
   - 邮件发送失败写入系统日志和操作审计。
   - 通知配置显示主管、工艺邮箱状态。
   - 增加清除本机桌面通知去重记录的按钮。

6. 部署与备份
   - 完善 Windows 局域网部署说明。
   - 增加数据库备份脚本或管理端备份按钮。
   - 管理端显示服务运行状态、监听目录、扫描间隔、最近扫描结果。

### 暂不做

- PDF 在线批注。
- Electron 桌面客户端。
- 坚果云 WebDAV/API 深度集成。
- 项目级动态审批流。
- 多级复杂权限。
- CAD 原始文件管理。

这些功能可以进入第三版或后续专项版本。

## 用户场景

### 场景 1：设计师提交了坏 PDF

1. 设计师放入 `.pdf` 文件。
2. 系统检测到文件头不是 `%PDF-`。
3. 审批单进入“PDF 无效”异常状态。
4. 管理员在异常列表中看到原因。
5. 管理员重新绑定或替换有效 PDF。
6. 系统重新进入待审核流程，并写入操作日志。

### 场景 2：审核中文件被删除

1. 文件在 `02-审批中` 被手动删除或坚果云同步丢失。
2. 系统标记审批单为 `file_missing`。
3. 待审核人不再看到该任务。
4. 管理员在异常列表中处理：重新绑定、重新提交或作废。

### 场景 3：提交了错误版本

1. 设计师提交了不应审核的版本。
2. 管理员在详情页点击“作废审批单”。
3. 系统保留记录，状态变为“已作废”。
4. 操作日志记录作废人、时间和原因。

### 场景 4：邮件通知不生效

1. 管理员配置 SMTP。
2. 点击“发送测试邮件”。
3. 页面显示成功或失败原因。
4. 失败详情写入系统日志，便于排查账号、端口、授权码问题。

## 数据模型设计

### approvals

现有审批表增加状态：

```text
voided
invalid_pdf
```

状态含义：

- `pending`：等待主管和工艺审核。
- `rejected`：已驳回。
- `approved_for_print`：已通过，待打印。
- `printed_archived`：已打印归档。
- `filename_invalid`：文件名异常。
- `file_missing`：文件丢失。
- `invalid_pdf`：PDF 内容无效。
- `voided`：已作废。

说明：

- 第二版可以继续保留 `filename_invalid`、`file_missing`，并新增 `invalid_pdf`、`voided`。
- `voided` 不参与审核队列，不参与待打印队列。
- `invalid_pdf` 不进入审核队列，必须替换有效 PDF 后才能重新提交审核。

### operation_logs

新增操作日志表：

```text
id
actor_user_id
actor_username
action
target_type
target_id
message
metadata_json
created_at
```

典型 action：

- `approval.created`
- `approval.reviewed`
- `approval.rejected`
- `approval.approved_for_print`
- `approval.printed`
- `approval.file_missing`
- `approval.invalid_pdf`
- `approval.rebound_file`
- `approval.voided`
- `system.scan_started`
- `system.scan_completed`
- `system.restart_requested`
- `mail.test_sent`
- `mail.send_failed`
- `user.created`
- `user.updated`
- `user.password_reset`

### scan_runs

新增扫描记录表：

```text
id
started_at
finished_at
status
processed_count
missing_count
invalid_count
error_message
triggered_by
```

用途：

- 管理端显示最近扫描结果。
- 重新扫描失败时保留错误信息。
- 便于判断坚果云同步目录是否长期异常。

## 后端接口设计

### 审批单修复

```text
POST /api/approvals/:id/void
POST /api/approvals/:id/rebind-file
POST /api/approvals/:id/retry-validation
```

说明：

- `void`：管理员作废审批单，必须填写原因。
- `rebind-file`：管理员选择服务器上的 PDF 路径，重新绑定当前审批单文件。
- `retry-validation`：重新检查 PDF 文件是否有效。

### 扫描

```text
POST /api/system/scan-now
GET /api/system/scan-runs
```

说明：

- `scan-now` 手动触发一次扫描。
- `scan-runs` 查询最近扫描记录。

### 操作日志

```text
GET /api/operation-logs
GET /api/approvals/:id/operation-logs
```

说明：

- 全局操作日志用于系统管理页。
- 审批单操作日志用于详情页时间线。

### 邮件测试

```text
POST /api/settings/test-smtp
```

输入：

```json
{
  "to": "test@example.com"
}
```

输出：

```json
{
  "ok": true
}
```

失败时返回可读错误，并写入操作日志。

## 文件处理规则

### 新文件进入系统前

检查顺序：

1. 文件扩展名是否 `.pdf`。
2. 文件是否稳定。
3. 文件名是否符合规则。
4. PDF 文件头是否有效。
5. 是否与已有项目、零件、版本重复。

处理结果：

- 文件名异常：创建 `filename_invalid` 记录。
- PDF 无效：创建 `invalid_pdf` 记录。
- 重复版本：不覆盖已存在有效审批单，记录日志并在扫描结果中提示。
- 正常文件：创建 `pending` 审批单并移动到 `02-审批中`。

### 重新绑定文件

管理员重新绑定文件时：

1. 选择服务器上的 PDF 文件。
2. 系统校验文件存在。
3. 校验 PDF 文件头。
4. 更新 `current_file_path`。
5. 状态恢复为 `pending` 或管理员指定的目标状态。
6. 记录操作日志。

第二版建议默认恢复为 `pending`，避免直接进入已通过或待打印。

## 权限设计

### admin

- 所有管理能力。
- 可作废、重新绑定、重新扫描、测试邮件、备份数据库。

### supervisor / process

- 正常审核。
- 可查看与自己相关的操作时间线。
- 不可修复文件、不可靠作废审批。

### printer

- 打印归档。
- 可查看待打印和已归档记录。

### designer

- 第二版仍保持预留。
- 暂不扩展设计师上传页面，继续以目录提交为主。

## 风险与取舍

### 风险

- SQLite 表约束迁移需要谨慎，避免破坏已有审批数据。
- 重新绑定文件涉及文件路径安全校验，必须限制在服务器可访问路径内。
- SMTP 失败原因可能来自公司网络、防火墙或授权码，系统只能提供可读错误和日志。

### 取舍

- 第二版不做 PDF 批注，避免引入重型前端依赖。
- 第二版不做坚果云 API，继续利用本地同步目录，降低账号和网络风险。
- 第二版不做项目级审核人配置，保留固定两人并行审核模型。
