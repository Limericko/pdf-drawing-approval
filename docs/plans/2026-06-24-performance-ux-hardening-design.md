# PDF 图纸审批系统性能与用户体验打磨方案

日期：2026-06-24

## 背景

系统已进入局域网部署和安装包分发阶段，核心审批流程、网页上传、签名、批注、运维追溯、更新发布都已具备。当前优化目标不是扩展复杂业务，而是让日常使用更快、更稳、更顺手，并为图纸数量增长后的排障提供依据。

## 目标

- 降低客户端首次进入和普通页面切换时的同步加载成本。
- 减少图纸台账搜索、详情页刷新、批注和签名操作中的重复请求。
- 优化常用页面的加载态、空态、错误提示、批量操作反馈和移动端可读性。
- 增加服务端慢接口观测，方便后续定位性能瓶颈。
- 复查数据库索引和长列表渲染策略，保持系统在更多图纸下可用。

## 推荐实现

本次同时覆盖三个方向，但以低风险、可验证为边界。

### 1. 前端性能

`App.tsx` 保留登录页和连接配置页同步加载，其余业务页面改为路由级 `React.lazy`。PDF 工作区已经异步加载，本次继续将大页面拆出首屏 bundle，并在侧边栏导航 hover/focus 时预加载目标页面，提升切换体感。

图纸台账关键词输入增加防抖和 `useDeferredValue`，输入框立即响应，后端请求只在用户停顿后发出。批量操作和筛选变化时保留当前反馈，避免页面闪烁。

### 2. 用户体验

统一业务页面加载状态：页面级懒加载显示明确的“正在打开...”提示；台账列表显示“正在刷新当前筛选结果”；详情页 PDF 检查失败时给出重试入口。

针对长列表和卡片面板增加 `content-visibility: auto`，让浏览器跳过屏幕外行的布局和绘制。对操作日志、批量结果、批注列表等可增长内容优先应用。

### 3. 后端长期性能

新增轻量请求耗时日志：超过阈值的 API 在服务端日志中输出 method、path、status、duration。该日志不影响响应，不记录密码、token 或请求体。

复查 SQLite 索引，保留当前幂等迁移方式。新增或调整索引只覆盖已上线查询路径：图纸状态筛选、签名状态筛选、项目零件版本追溯、提交人台账、操作日志按审批单读取。

## 非目标

- 不改变固定主管和固定工艺并行审核流程。
- 不引入新的前端状态管理库。
- 不把 PDF 预览替换为完整 PDF.js Viewer。
- 不做虚拟滚动库接入；当前分页和 `content-visibility` 足够覆盖这一轮。
- 不做自动静默更新安装。

## 风险与处理

- 路由懒加载可能影响页面测试和导入路径。处理：保留类型导入，新增源级测试覆盖 `lazy`、`Suspense` 和预加载入口。
- 搜索防抖可能让用户感觉结果稍有延迟。处理：输入立即更新，列表区显示刷新提示，停顿后请求。
- 慢接口日志可能产生噪音。处理：默认阈值较高，只记录 API 路径和耗时，不记录敏感内容。
- 新增索引会增加少量写入成本。处理：只添加高价值索引，使用 `CREATE INDEX IF NOT EXISTS`，不做破坏性 schema 变更。

## 验收标准

- `npm run build` 后主入口 chunk 不再同步包含所有业务页面代码。
- 图纸台账搜索输入不会每个按键立刻请求后端。
- 页面懒加载和列表刷新都有清楚的中文反馈。
- 服务端慢 API 有可读日志，默认不泄露敏感内容。
- 数据库索引测试通过。
- `npm test`、`npm run build`、`npm run desktop:test` 通过。
