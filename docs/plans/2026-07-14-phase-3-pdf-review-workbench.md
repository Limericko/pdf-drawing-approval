# Phase 3 PDF 审阅与标注工作台实施计划

- 日期：2026-07-14
- 状态：已完成
- 分支：`codex/phase-3-pdf-review-workbench`
- 前置门禁：Phase 0、Phase 1、Phase 2 已完成
- 产品方向：文档优先三栏式、混合批注、精密工业、高密度、安静、可扫描

## 目标

把现有审批详情页重构为完整 PDF Studio，而不是只调整布局。主管、工艺和设计师必须能在同一工作面完成查看、标注、正式问题处理、复核关闭和审批；未关闭的高严重级问题必须由前后端共同阻止审批通过。

## 已确认边界

- 桌面 `>=1281px`：112px 缩略图栏、自适应文档画布、320px 检查器三栏。
- 紧凑桌面 `981–1280px`：88px 缩略图栏、自适应画布、280px 检查器。
- `681–980px`：画布优先，缩略图和检查器改为 Drawer。
- `<=680px`：支持 PDF 查看、问题定位、讨论、状态处理和审核结论；精确绘制不是主要入口。
- 保留比例坐标、现有 PDF.js 渲染、审查版 PDF 与正式签后 PDF 隔离，不引入第三方 UI/PDF 编辑框架。
- 普通说明继续使用批注；正式问题拥有严重级、负责人、到期时间和独立状态机。
- 正式问题状态：`待处理 -> 处理中 -> 待复核 -> 已关闭`。待复核可被退回处理中。
- 处理人不能关闭自己的问题；创建人、主管/工艺审核角色或管理员可复核。管理员绕过正常复核强制关闭必须填写原因并写审计。

## Task 1：现状审计和验收矩阵

1. 固化现有批注、评论、审批、PDF、签名和 PDM 调用点。
2. 为三栏、工具状态、问题闭环、阻断规则、长文档和响应式建立可验证清单。
3. 建立 Phase 3 计划文档和 RED 测试入口。

退出条件：每项规格都有代码落点和测试证据，不能把旧 `kind=issue + resolved` 当作完整问题模型。

## Task 2：正式问题领域模型

新增独立 `approval_issues` 与 `approval_issue_events`：

- 问题关联 approval，可选关联 annotation。
- 字段包含标题、说明、严重级、负责人、状态、到期时间、创建人、处理说明、复核说明、强制关闭原因及完整时间戳。
- 事件记录每次创建、领取、提交复核、退回、关闭和强制关闭。
- Repository 提供列表、详情、创建、状态迁移、阻断计数和删除审批级联清理。
- 旧 `approval_comments.kind=issue` 保持只读兼容；新正式问题只写入新表。

退出条件：数据库迁移幂等；Repository 状态机和权限前置条件单测全部通过。

## Task 3：问题 API、权限与审计

新增审批下的问题路由：

- `GET /api/approvals/:id/issues`
- `POST /api/approvals/:id/issues`
- `PATCH /api/approvals/:id/issues/:issueId`
- `POST /api/approvals/:id/issues/:issueId/transitions`

要求：

- 创建者为主管、工艺或管理员；负责人必须是有效用户。
- 负责人/设计师可领取并提交待复核，处理说明必填。
- 创建人、主管、工艺或管理员可退回/关闭；负责人不能关闭自己的问题。
- 管理员强制关闭必须填写原因。
- 所有动作写 `approval_issue_events` 和 operation log。

退出条件：正常、越权、非法跳转、重复提交、无效负责人和审计测试全部通过。

## Task 4：高严重级审批阻断

- 审批 `approved` 前查询未关闭的 high/critical 问题。
- 后端返回稳定错误 `OPEN_HIGH_SEVERITY_ISSUES` 和数量，不移动文件、不改变审核状态。
- 前端审核动作栏显示阻断原因并定位问题检查器。
- 中低严重级显示明确提醒，但按既有权限允许继续。

退出条件：主管、工艺、管理员路径在路由测试中一致；前端不允许只靠禁用按钮隐藏原因。

## Task 6：文档优先三栏工作台

- 左栏显示缩略图、页码、每页批注/问题数量和当前页。
- 中栏固定顶栏工具、可滚动画布和底部状态栏。
- 右栏使用“问题 / 说明 / 属性 / 审核”稳定标签，不再使用可拖动浮动协同窗。
- 顶栏显示图号、项目、版本、只读/保存状态、撤销/重做。
- 底栏显示保存状态、页码、缩放、问题阻断摘要和审核动作。
- “通过并打开下一张”保留，但不破坏三栏布局。

退出条件：旧详情页同职责 DOM 和全局样式零引用后删除。

## Task 7：画布上下文编辑与混合模式

- 绘制后在图形附近打开上下文编辑器。
- 用户选择“说明”或“正式问题”。
- 说明保存为 annotation；正式问题创建 annotation + issue 关联。
- 正式问题表单包含严重级、负责人和到期时间。
- 右侧列表可按状态、严重级、负责人、页码筛选并定位画布。
- 检查器支持领取、提交复核、退回、关闭和管理员强制关闭。

退出条件：取消不产生孤立数据；失败时保留草稿并提供重试；旧问题记录仍可查看。

## Task 8：编辑效率和性能

- 撤销/重做覆盖当前会话的创建和几何调整，服务端操作失败时回滚 UI。
- 连续标注保留工具和颜色。
- 支持复制、删除、移动、缩放和改色。
- 缩略图低分辨率延迟渲染，画布页面按可视区/邻近页加载。
- 长文档验证不会一次渲染全部高清 canvas。
- 显示保存中、已保存、保存失败和重试。

退出条件：非空像素、缩放、跳页、长文档渲染数和撤销/重做真实浏览器测试通过。

## Task 9：响应式与可访问性

- `>=981px` 支持精确绘制和可调整分栏。
- `681–980px` 使用缩略图/检查器 Drawer。
- `<=680px` 仍可查看 PDF、定位问题、讨论、改变问题状态和审核；底部动作栏触控目标至少 44px。
- 所有图标按钮有可访问名称；状态不只靠颜色；快捷键不拦截输入框。

退出条件：1440×900、1280×800、1024×768、768×1024、390×844 全部通过截图、axe、溢出和键盘门禁。

## Task 10：最终闭环与门禁

真实角色闭环：

1. 主管创建高严重级正式问题并指派设计师。
2. 高严重级未关闭时，主管/工艺审批通过被阻断。
3. 设计师领取、处理并提交待复核。
4. 创建人退回一次，设计师修订后再次提交。
5. 创建人关闭，审批恢复可通过。

```powershell

npm run e2e:typecheck
npm run build

npm run e2e:ui
npm run e2e
npm run e2e:platform
git diff --check
```
