# PDF 图纸审批系统第三版方案

日期：2026-06-16

## 版本定位

第三版目标是把第二版从“可日常运营、可纠错、可追溯的审批工具”升级为“可正式上线使用的签审闭环系统”。

第三版同时做三件事：

1. 正式上线运维增强。
2. 协同追溯增强。
3. 网页提交与可视手写签名盖章。

本版本继续保持局域网内部使用、Windows 优先、坚果云本地同步目录为文件底座，不引入坚果云 API、云存储或复杂审批流引擎。

## 已确认决策

- 签名类型采用可视手写签名盖章，不做 CA/证书数字签名。
- 设计师通过网页上传 PDF 作为主要提交入口。
- 网页上传后仍写入坚果云标准目录，保持文件可见、可同步、可备份。
- 保留目录监听和手动扫描作为兜底入口。
- 设计师提交时一次性放置设计、主管、工艺三个签名框。
- 签名框在 PDF 预览上拖拽定位，并支持缩放调整。
- 主管和工艺只审核，不调整签名位置。
- 主管和工艺都通过后，系统自动把设计师、主管、工艺的签名盖到 PDF 对应位置。
- 签后 PDF 生成新文件，不覆盖原审批 PDF。
- 打印使用签后 PDF。

## 设计原则

- 不推翻第二版，增量扩展提交、签名、追溯和运维能力。
- 原始 PDF、签后 PDF、审批记录、签名位置和操作日志都应可追溯。
- 签名失败不能静默失败，必须有异常状态和重试入口。
- 网页上传要降低设计师操作成本，但不能破坏坚果云标准目录管理方式。
- 正式上线能力优先覆盖备份、诊断、日志、恢复说明和 Windows 服务运行稳定性。

## 目标用户与角色

### 设计师

- 网页上传待审 PDF。
- 确认项目、零件名、版本。
- 配置或使用自己的手写签名。
- 在 PDF 上放置设计、主管、工艺三个签名框。
- 查看提交后的审批进度和驳回意见。

### 主管

- 查看待审图纸。
- 通过或驳回。
- 使用自己预设的手写签名参与自动盖章。

### 工艺

- 查看待审图纸。
- 通过或驳回。
- 使用自己预设的手写签名参与自动盖章。

### 管理员

- 管理用户和签名配置状态。
- 处理异常审批单。
- 查看操作日志、服务日志、诊断结果。
- 触发扫描、备份、签名重试和报表导出。

### 打印人员

- 查看已通过待打印图纸。
- 打开或打印签后 PDF。
- 标记已打印归档。

## 核心流程

### 网页上传提交流程

1. 设计师登录系统，进入“提交图纸”。
2. 上传 PDF 文件。
3. 系统读取文件名，尝试解析 `零件名-a0A0.pdf`。
4. 页面自动填充零件名和版本，项目名由设计师选择或填写。
5. 设计师可修正项目、零件名和版本。
6. 系统按标准命名生成最终文件名。
7. 系统校验 PDF 文件头、版本重复和必填字段。
8. 页面加载 PDF 预览。
9. 设计师拖拽放置三个签名框：设计、主管、工艺。
10. 设计师提交审批。
11. 系统把 PDF 写入坚果云标准目录并创建审批单。
12. 审批单进入主管和工艺并行审核。

### 自动签名流程

1. 主管通过。
2. 工艺通过。
3. 系统确认审批单已经满足签名生成条件：
   - 审批状态为已通过待打印。
   - 设计、主管、工艺三类签名框齐全。
   - 三个角色都有可用签名图片。
   - 原始审批 PDF 文件存在且有效。
4. 系统读取原始审批 PDF。
5. 系统把三个签名图片盖到对应页码和坐标。
6. 系统生成签后 PDF 新文件。
7. 系统记录签后文件路径、生成时间、文件哈希和操作日志。
8. 待打印列表默认打开签后 PDF。

### 目录兜底流程

第二版已有目录监听继续保留：

- `01-待提交` 下新增 PDF 仍可创建审批单。
- 根目录或项目目录放入 PDF 仍可按现有规则处理。
- watcher 漏事件时继续由定时扫描补偿。

目录提交的审批单如果没有签名框，不自动进入完整签审闭环。第三版建议将其标记为“待补签名位置”，由管理员或设计师在详情页补齐签名框后再继续审批。

## 文件目录设计

继续沿用标准目录：

```text
审批根目录
├─ 01-待提交
├─ 02-审批中
├─ 03-已驳回
├─ 04-已通过待打印
└─ 05-已打印归档
```

网页上传写入规则：

- 上传暂存文件先落到服务端临时目录。
- 提交确认后写入 `02-审批中\项目名\标准文件名.pdf`。
- 如果需要保留上传原始名，可写入审批记录，不作为正式流转文件名。

签后 PDF 输出规则：

```text
04-已通过待打印\项目名\零件名-a0A0-签审.pdf
```

如同名文件已存在，系统追加审批单编号或时间戳，避免覆盖。

打印归档时：

- 签后 PDF 移动到 `05-已打印归档\项目名\`。
- 原始审批 PDF 不覆盖、不删除。
- 数据库保留原始路径和签后路径。

## 签名资产设计

### 签名来源

每个用户支持两种方式配置签名：

- 上传 PNG 签名图片。
- 在线手写后保存为 PNG。

第三版只支持透明背景或白底签名图片，前端应提示用户使用清晰深色签名。

### 签名归属

签名资产绑定用户：

- 设计师使用提交人的签名。
- 主管使用主管审核人的签名。
- 工艺使用工艺审核人的签名。

固定审核人仍保持第二版模式。后续如扩展项目级审核人，再调整签名人解析规则。

### 签名安全边界

第三版签名是内部流程确认用途，不等同法律意义上的数字签名。

系统应记录：

- 使用了哪个用户的签名。
- 签名生成时间。
- 签名前原始 PDF 哈希。
- 签后 PDF 哈希。
- 触发签名生成的审批单和操作日志。

管理员可以查看签名配置状态，但不默认代替别人上传签名。若需要管理员代管签名，应作为后续明确权限功能处理。

## 签名框定位设计

签名框字段：

- 审批单 ID。
- 签名角色：`designer`、`supervisor`、`process`。
- PDF 页码。
- x 坐标。
- y 坐标。
- 宽度。
- 高度。

坐标保存为 PDF 页面坐标或页面归一化比例，不能保存浏览器屏幕像素。推荐保存归一化比例：

```text
x_ratio
y_ratio
width_ratio
height_ratio
```

生成签后 PDF 时再根据实际页面宽高换算为 PDF 坐标。这样可以降低不同浏览器缩放、不同 PDF 尺寸导致的错位风险。

前端交互：

- PDF 预览区域显示签名框。
- 三个签名框有不同标签：设计、主管、工艺。
- 支持拖动。
- 支持四角或边缘缩放。
- 支持选择页码。
- 支持重置到默认位置。
- 提交前必须校验三类签名框齐全。

## 状态设计

沿用第二版状态：

- `pending`
- `rejected`
- `approved_for_print`
- `printed_archived`
- `filename_invalid`
- `file_missing`
- `invalid_pdf`
- `voided`

第三版新增签名相关状态建议使用独立字段 `signature_status`，避免审批主状态膨胀：

- `not_required`：不需要签名。
- `placement_required`：缺少签名框。
- `pending`：等待审批完成后签名。
- `ready`：签名条件齐全，待生成。
- `generated`：签后 PDF 已生成。
- `failed`：签名生成失败。

典型组合：

- 审批中：`status = pending`，`signature_status = pending`。
- 已通过但签名失败：`status = approved_for_print`，`signature_status = failed`。
- 已通过且签后 PDF 已生成：`status = approved_for_print`，`signature_status = generated`。
- 目录兜底提交且未定位：`status = pending` 或异常处理前状态，`signature_status = placement_required`。

## 数据模型设计

### approvals 扩展字段

建议新增：

```text
submitted_by_user_id
source
original_file_hash
signed_file_path
signed_file_hash
signed_at
signature_status

```

字段说明：

- `source`：`web_upload` 或 `folder_watch`。
- `original_file_hash`：原始审批 PDF 哈希。
- `signed_file_path`：签后 PDF 路径。
- `signed_file_hash`：签后 PDF 哈希。
- `signature_error`：最近一次签名失败原因。

### signature_assets

```text

user_id

file_path

```

`kind` 可取：

- `uploaded_png`


## 后端接口设计

### 网页上传

```text
POST /api/submissions/upload
POST /api/submissions
```

`upload` 用于上传临时 PDF 并返回解析结果。

`submissions` 用于确认提交：

```json
{
  "uploadId": "tmp-xxx",
  "projectName": "项目A",
  "partName": "零件名",
  "version": "a0A0",
  "placements": [
    { "role": "designer", "pageNumber": 1, "xRatio": 0.72, "yRatio": 0.82, "widthRatio": 0.12, "heightRatio": 0.05 },
    { "role": "supervisor", "pageNumber": 1, "xRatio": 0.82, "yRatio": 0.82, "widthRatio": 0.12, "heightRatio": 0.05 },
    { "role": "process", "pageNumber": 1, "xRatio": 0.92, "yRatio": 0.82, "widthRatio": 0.12, "heightRatio": 0.05 }
  ]
}
```

### 签名资产

```text
GET /api/signatures/me
POST /api/signatures/me/upload
POST /api/signatures/me/draw
GET /api/signatures/status
```

`/api/signatures/status` 管理员使用，显示用户签名配置状态。

### 签名生成

```text
POST /api/approvals/:id/generate-signed-pdf

```

自动生成由审核通过后触发。手动接口用于失败后重试。

### 评论与问题

```text
GET /api/approvals/:id/comments
POST /api/approvals/:id/comments
POST /api/approvals/:id/comments/:commentId/resolve
```

### 追溯导出

```text
GET /api/reports/approvals.csv
```

支持查询参数：

```text

status
from
to
```

导出字段：

- 审批单 ID。
- 项目。
- 零件。
- 版本。
- 状态。
- 提交人。
- 提交时间。
- 主管审核状态和时间。
- 工艺审核状态和时间。
- 签名状态。
- 签后文件路径。
- 原始文件哈希。
- 签后文件哈希。
- 打印归档时间。
- 最近问题/评论摘要。

### 运维诊断

```text
GET /api/system/diagnostics
POST /api/system/backup
GET /api/system/backups
```

诊断项：

- 数据库是否可读写。
- 审批根目录是否存在。
- 标准目录是否齐全。
- `02-审批中`、`04-已通过待打印`、`05-已打印归档` 是否可写。
- 最近扫描时间和结果。
- 最近备份时间和结果。
- 日志文件是否可读取。
- 当前服务启动时间。

## PDF 签名技术选择

建议引入成熟 PDF 库完成图片盖章。

优先候选：

- `pdf-lib`：纯 JS，适合 Node 环境盖 PNG 到 PDF 页面，避免 Windows 原生依赖。

不建议：

- 需要本机编译的 PDF 原生模块。
- 依赖外部服务的 PDF 处理 API。
- 浏览器端直接生成最终签后 PDF。

理由：

- 当前项目已经明确避免 `better-sqlite3` 这类 Windows 原生编译风险。
- 服务端生成可统一记录日志、哈希和失败原因。
- 局域网部署不应依赖外网 PDF 服务。

## 错误处理

### 上传错误

- 非 PDF：拒绝提交。
- PDF 文件头无效：拒绝提交并显示中文原因。
- 文件名无法解析：允许设计师手动填写后提交。
- 版本重复：拒绝提交，提示已有审批单。

### 签名错误

- 缺少签名资产：`signature_status = failed`，错误说明缺少哪个用户签名。
- 缺少签名框：`signature_status = placement_required`。
- 原始文件丢失：保持或转入 `file_missing`。
- PDF 写入失败：记录错误，允许管理员重试。

### 文件输出错误

- 目标文件已存在：自动生成不冲突的新文件名。
- 目标目录不可写：写入诊断失败项和操作日志。

- 设计师提交时可在 PDF 预览上放置设计、主管、工艺三个签名框。
- 签名框支持拖动和缩放。
- 每个用户可配置自己的手写签名图片。
- 主管和工艺通过后自动生成签后 PDF。
- 签后 PDF 不覆盖原审批 PDF。
- 待打印默认使用签后 PDF。
- 签名失败有明确状态、日志和重试入口。
- 评论/问题记录可追溯。
- 可导出审批追溯 CSV。
- 管理端可查看运行诊断、备份记录、服务日志和操作日志。
- `npm test` 通过。
- `npm run build` 通过。
- 完成一次端到端现场验证。

## 风险与取舍

### 风险

- PDF 页面坐标与浏览器预览坐标可能存在换算误差，必须通过测试 PDF 验证。
- 在线手写签名质量取决于鼠标或触控板，可能需要用户多次调整。
- 签名图片透明背景处理如果做得过重会增加复杂度，第三版先支持 PNG 原样盖章。
- 自动签名生成涉及文件写入，必须重点验证坚果云目录权限和文件占用。
- 数据库扩展字段较多，迁移需保持兼容已有 V1/V2 数据。

### 取舍

- 第三版不做 CA 数字签名。
- 第三版不做 PDF 在线批注。
- 第三版不做动态多级审批流。
- 第三版不做坚果云 API 集成。
- 第三版不做管理员代签能力，除非后续明确授权边界。
