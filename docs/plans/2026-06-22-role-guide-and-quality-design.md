# 角色流程向导与全量质量检查设计

## 目标

在不改变局域网部署架构的前提下，处理本次全量检查发现的真实问题，并在用户登录后提供按角色区分的流程向导。旧 `printer` 打印角色正式从当前流程中移除。

## 检查结论

- 自动化基线通过：`npm test`、`npm run build`、`npm run desktop:test` 均成功。
- `npm audit` 使用 npm 镜像源时失败，原因是镜像不支持 npm security audit 接口。
- 使用官方 npm registry 审计发现生产依赖 `nodemailer@6.9.16` 存在 high 级安全公告，需升级并回归邮件发送测试。
- 提交图纸页批量上传时，历史版本查询按每个文件发起请求。多个文件同零件名时会重复查询。
- 全部图纸页切换筛选较快时，旧请求结果存在覆盖新筛选结果的风险。
- 旧 `printer` 角色已经不是当前业务流程的一部分，应从前端类型、导航和展示逻辑中移除；后端新建/修改用户接口已经不允许 `printer`，保留数据库旧 CHECK 仅用于历史数据兼容。

## 方案

### 安全依赖

升级 `nodemailer` 到审计建议的安全版本。由于是跨主版本升级，必须验证：

- 邮件通知 helper 测试。
- 审批创建邮件通知测试。
- 全量测试。
- 生产构建。

### 性能与稳定性

提交图纸页新增纯函数构建历史版本查询计划：

- 按项目名和零件名去重。
- 忽略空项目或空零件。
- 保留每个查询命中的所有 `clientId`，让一个请求结果可回填多个批量文件。

同时给查询 effect 增加短防抖，减少输入过程中频繁请求。

全部图纸页请求 effect 增加 active guard。筛选变化后，旧请求完成时不再写入列表或错误状态。

### 角色流程向导

新增紧凑的顶部流程向导组件，登录后显示在主内容上方：

- 设计师：配置签名、上传 PDF、放置签名框、提交审批、通过后打印归档。
- 主管：进入待我审核、查看图纸与评论、通过或驳回。
- 工艺：进入待我审核、检查工艺可制造性、通过或驳回。
- 管理员：配置目录、维护用户与签名、查看日志风险、备份维护。

向导按角色提供主行动按钮，收起状态写入本机 `localStorage`。不写数据库，不影响其他电脑。

## 非目标

- 不新增旧打印角色入口。
- 不注册 Windows Service。
- 不改审批数据库主流程。
- 不改固定主管和固定工艺并行审核模型。
