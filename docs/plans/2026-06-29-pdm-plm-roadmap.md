# PDF 审批系统向轻量 PDM / PLM 演进路线图

日期：2026-06-29  
当前系统版本基线：0.9.1  
当前状态：已上线使用  
适用范围：公司局域网、Windows 客户端、Windows 服务端、小型机械设计团队

## 1. 结论

当前系统可以向 PDM 方向发展，但不建议直接定位为完整 PLM。

更合理的产品定位是：

```text
PDF 图纸审批系统
  -> 轻量图纸 PDM
  -> 工程变更与 BOM 管理
  -> 按需扩展到 PLM 能力
```

近期目标应是“轻量图纸 PDM”，而不是“大而全 PLM”。原因是当前系统已经掌握了图纸文件、版本、审批、签名、归档和追溯，但还没有完整建立零件主数据、产品结构、工程变更、BOM 和跨部门生命周期流程。

## 2. 第一性原理

PDM / PLM 的根本目的不是增加审批页面，而是让产品相关信息在生命周期内保持：

- 唯一：同一个零件、图纸、版本不能有多个互相冲突的来源。
- 可控：谁能创建、修改、审核、发布和作废必须明确。
- 可追溯：任何版本变化都能回答谁、何时、为什么、改了什么。
- 可复用：历史图纸、零件、BOM、变更记录能被后续项目可靠引用。
- 可冻结：已发布版本不能被随意覆盖，只能通过新版本或变更流程演进。

从这个角度看，当前系统的本质是：

```text
图纸文件审批与受控发布系统
```

PDM 的本质是：

```text
产品数据受控系统
```

PLM 的本质是：

```text
产品全生命周期协同系统
```

因此，当前系统离轻量 PDM 很近，离完整 PLM 还有明显距离。

## 3. 当前系统已有基础

当前系统已经具备轻量 PDM 的一部分底座：

- 图纸文件上传和目录投递。
- 图纸版本命名约定，例如 `零件名-a0A0.pdf`。
- 主管和工艺并行审批。
- 设计、主管、工艺三方手写签名。
- 签后 PDF 与审查版 PDF。
- 图纸批注、评论、时间线。
- 打印归档。
- 用户、角色、权限。
- 操作日志、服务日志、风险看板。
- 备份、恢复说明、自动维护。
- 局域网安装包、客户端更新、服务端发布目录。

这些能力说明系统已经从“临时审批工具”进入“受控图纸流程工具”阶段。

## 4. 当前关键缺口

要成为 PDM，系统还缺少几个核心对象。

### 4.1 零件主数据

当前系统主要围绕 PDF 审批单工作，还没有独立的“零件档案”。

后续需要建立：

- 零件号。
- 零件名称。
- 所属项目。
- 材料、规格、重量等基础属性。
- 当前有效图纸版本。
- 生命周期状态。
- 历史版本列表。

### 4.2 图纸版本库

当前系统能识别文件名版本，但还需要从“审批记录版本”升级为“图纸版本库”。

后续需要明确：

- 一个零件可关联多个图纸版本。
- 当前有效版本只能有一个。
- 旧版本可查，但不能误用。
- 原始 PDF、审查版 PDF、签后 PDF、CAD 源文件都归属于同一个图纸版本。
- 版本发布后不能覆盖，只能作废或产生新版本。

### 4.3 工程变更单

PDM 的核心不是只存版本，而是解释版本为什么变化。

后续需要建立：

- 变更申请。
- 变更原因。
- 影响零件和图纸。
- 旧版本和新版本。
- 审批结论。
- 发布记录。

### 4.4 BOM / 产品结构

完整 PDM 通常需要 BOM，但不建议第一步就做复杂 BOM。

后续可逐步建立：

- 产品。
- 部件。
- 零件。
- 数量。
- 单位。
- 版本。
- 替代件、借用件。
- Excel 导入导出。

## 5. 演进路线

### 阶段 0：生产稳定期

目标：保护当前已上线系统稳定运行。

重点：

- 不做大规模重构。
- 不破坏当前审批、签名、打印归档主流程。
- 每次发布都保留更新日志、测试记录和回滚说明。
- 加强备份、恢复、诊断、日志导出。

建议版本方向：

```text
0.9.x 生产运维增强
```

退出条件：

- 团队能稳定完成真实图纸审批。
- 管理员能独立处理连接、上传、签名、打印、更新、备份等常见问题。
- 至少完成一次备份恢复演练。

### 阶段 1：轻量图纸 PDM V1

目标：从“审批单中心”升级为“零件和图纸版本中心”。

建议范围：

- 零件档案。
- 图纸版本库。
- 当前有效版本。
- 历史版本链。
- 图纸版本与审批记录关联。
- 图纸版本与签后 PDF、审查版 PDF、原始 PDF 关联。
- 按零件、项目、版本搜索。

不建议在此阶段加入复杂 BOM、ERP 集成或多部门生命周期。

建议版本方向：

```text
1.0 轻量图纸 PDM
```

验收标准：

- 用户能按零件找到所有历史图纸版本。
- 用户能明确看到当前有效版本。
- 已发布版本不能被直接覆盖。
- 任何版本变更都能追溯到审批记录。

### 阶段 2：工程变更管理

目标：解释版本变化，避免“只知道换了版本，不知道为什么换”。

建议范围：

- 变更单编号。
- 变更原因。
- 影响范围。
- 关联零件和图纸版本。
- 变更前版本。
- 变更后版本。
- 变更审批。
- 变更发布。

建议版本方向：

```text
1.1 工程变更管理
```

验收标准：

- 每次关键图纸变更都能关联一个变更单。
- 可查询某个零件为什么从旧版变为新版。
- 可导出变更记录用于质量或客户追溯。

### 阶段 3：轻量 BOM

目标：建立产品结构，但先保持简单。

建议范围：

- 产品档案。
- 产品下的部件和零件层级。
- 数量、单位、备注。
- 每个零件关联当前有效图纸版本。
- Excel 导入导出。
- BOM 快照。

建议版本方向：

```text
1.2 轻量 BOM
```

验收标准：

- 能查看一个产品由哪些零件组成。
- 能看到每个零件对应的当前有效图纸版本。
- 能导出 BOM 给采购、工艺或生产使用。

### 阶段 4：受控文件库

目标：从“文件夹流转”进一步升级为“受控文件库”。

建议范围：

- CAD 源文件上传。
- 附件管理。
- 文件哈希。
- 文件冻结。
- 签出 / 签入。
- 文件替换必须走新版本或变更。

建议版本方向：

```text
1.3 受控文件库
```

验收标准：

- 发布后的文件不能被直接覆盖。
- CAD 源文件、PDF、签后 PDF 能作为同一图纸版本的不同文件类型管理。
- 管理员能查到文件是否被替换、何时替换、由谁替换。

### 阶段 5：PLM 扩展

目标：从工程数据管理扩展到产品生命周期协同。

只有在轻量 PDM 稳定后，才考虑 PLM。

可能范围：

- 项目阶段管理。
- 工艺路线。
- 质量问题闭环。
- 客户需求和设计任务。
- 供应商资料。
- ERP / MES / 采购系统集成。
- 成本、库存、生产状态联动。

建议版本方向：

```text
2.x PLM 扩展
```

进入条件：

- 零件档案、图纸版本库、变更单、BOM 已稳定使用。
- 团队已经形成统一图号、零件号、版本、变更规则。
- 有明确跨部门流程需求，而不是单纯想增加功能。

## 6. PDM V1 推荐范围

下一阶段如果启动 PDM，建议只做以下内容。

### 6.1 数据对象

新增核心对象：

- Part：零件档案。
- DrawingRevision：图纸版本。
- ReleaseState：版本状态。
- PartRevisionLink：零件与图纸版本关系。

暂不引入复杂对象：

- 完整产品 BOM。
- 多级工艺路线。
- ERP 物料同步。
- 供应商数据。
- 成本数据。

### 6.2 页面入口

建议新增：

- 零件档案列表。
- 零件详情。
- 图纸版本列表。
- 图纸版本详情。
- 当前有效版本标记。
- 历史版本链。

现有页面保留：

- 提交图纸。
- 待我审核。
- 全部图纸。
- 审批详情。
- 系统管理。

### 6.3 和当前审批流程的关系

不要推翻当前审批流程。

推荐方式：

1. 设计师仍从“提交图纸”上传。
2. 系统根据项目、零件名、版本尝试匹配或创建零件档案。
3. 审批通过并签名成功后，生成一个已发布图纸版本。
4. 已发布图纸版本成为该零件的当前有效版本。
5. 旧版本自动进入历史版本。

这样可以平滑演进，不打断当前已上线流程。

## 7. 明确暂不做

近期不建议做：

- 完整 PLM。
- 复杂组织架构和多级权限。
- ERP / MES 深度集成。
- 多站点部署。
- 云端账号体系。
- 全量 CAD 集成。
- 自动解析 CAD BOM。
- 复杂流程引擎。

原因：

- 当前系统已上线，应优先稳定。
- 小团队的真实收益主要来自图纸版本受控和追溯。
- 过早做大功能会增加运维、培训、数据迁移和误操作风险。

## 8. 关键决策门槛

在从阶段 0 进入阶段 1 前，需要确认：

- 团队是否已经统一零件号或图号规则。
- 当前文件名 `零件名-版本.pdf` 是否足以作为过渡规则。
- 是否需要区分零件号和图号。
- CAD 源文件是否要纳入首版 PDM。
- 当前有效版本由谁发布。
- 旧版本是否允许普通用户下载。
- 作废图纸是否需要审批。

在从阶段 1 进入阶段 2 前，需要确认：

- 哪些变更必须走变更单。
- 小版本变更和大版本变更如何区分。
- 变更单是否需要主管、工艺、质量或生产参与。
- 变更发布后是否通知相关人员。

在从阶段 2 进入阶段 3 前，需要确认：

- BOM 的维护责任人是谁。
- BOM 是否以设计为准，还是需要和生产/采购核对。
- 是否已有 Excel BOM 模板。
- 是否要区分设计 BOM、工艺 BOM、生产 BOM。

## 9. 技术演进建议

### 9.1 当前阶段继续保留

当前 5-10 人局域网使用，建议继续保留：

- Windows-first。
- 局域网部署。
- Electron 客户端。
- Electron 服务端安装包。
- 内置 SQLite。
- 坚果云目录兼容。
- 当前更新发布目录。

### 9.2 何时考虑数据库升级

如果出现以下情况，再考虑从 SQLite 升级到 PostgreSQL 或 SQL Server：

- 用户超过 30 人。
- 同时访问明显变多。
- BOM 和变更数据量明显上升。
- 需要更强的并发写入。
- 需要跨部门长期报表。
- 需要和 ERP / MES 做稳定集成。

不要在 PDM V1 一开始就升级数据库。先验证业务模型更重要。

### 9.3 文件库演进

短期：

- 保留当前审批根目录和状态目录。
- 继续兼容网页上传和目录投递。
- 发布后的签后 PDF 继续归档。

中期：

- 为每个图纸版本建立受控文件记录。
- 对原始 PDF、审查版 PDF、签后 PDF、CAD 源文件记录哈希。
- 禁止直接覆盖已发布文件。

长期：

- 建立独立受控文件库。
- 目录结构不再作为业务状态唯一来源。
- 文件夹只作为存储载体，业务状态以数据库为准。

## 10. 风险清单

### 10.1 业务风险

- 没有统一零件号规则，会导致零件档案混乱。
- 只按文件名识别版本，可能无法支撑复杂产品。
- 变更流程定义不清，会让版本发布失去约束。
- BOM 责任不清，会出现多个版本的 BOM。

### 10.2 技术风险

- 已上线系统不宜大规模改造审批主流程。
- 版本库引入后，需要谨慎处理历史审批数据迁移。
- 文件冻结和作废规则必须防止误删、误覆盖。
- 如果过早引入复杂 PLM 对象，会增加维护成本。

### 10.3 运维风险

- 当前安装包未企业代码签名，正式推广时 Windows 安全提醒仍存在。
- 服务端升级前必须备份。
- 发布目录仍需维护真实运行路径 `E:\PDF服务端\pdf-approval\releases`。
- 后续任何 PDM 数据结构变更都必须有迁移和回滚说明。

## 11. 推荐下一步

下一步建议先做设计，不直接编码。

建议文档：

```text
docs/plans/YYYY-MM-DD-pdm-v1-design.md
```

建议设计主题：

```text
PDM V1：零件档案 + 图纸版本库
```

优先明确：

- 零件号规则。
- 图号和零件号是否分离。
- 图纸版本状态。
- 当前有效版本规则。
- 审批通过后如何生成图纸版本。
- 历史审批数据如何映射到零件和图纸版本。

只有这些规则稳定后，再进入实现计划。
